### `-Group Server`/`-Group ArcGIS` — แยก Server กับ ArcGIS ออกจากกันเด็ดขาด (23 ก.ค. 2569)

ผู้ใช้ขอให้แยก Server กับ ArcGIS ออกจากกัน **ไม่ให้รันพร้อมกัน** เพราะบางเครื่องต้องการแค่รายละเอียด
Server เท่านั้น ไม่เกี่ยวกับ ArcGIS เลย เพิ่มพารามิเตอร์ `-Group Server|ArcGIS` ใน `Start-PMCheck.ps1`
และหน้าจอเลือกโหมดใหม่เป็นหน้าแรกของเมนู (`Show-PMMenu.ps1`)

**กติกาที่บังคับใน `Start-PMCheck.ps1`** (ใช้ `Id -like 'AGS*'` แยกกลุ่ม ครอบคลุม `AGSDATA`/`AGSLOG`
ในอนาคตโดยอัตโนมัติ ไม่ต้องแก้โค้ดเพิ่มเมื่อมีหัวข้อ ArcGIS ใหม่)

- `-Only` ที่ผสมหัวข้อ Server กับ ArcGIS (เช่น `-Only DISK,AGS`) **ถูกปฏิเสธเสมอ** ไม่ว่าจะระบุ
  `-Group` หรือไม่ — กติกานี้ไม่ผูกกับ `-Group` เพื่อจับกรณี CLI ผสมมือด้วย
- `-Group ArcGIS` เท่ากับขอกลุ่มนั้นตรง ๆ จึง **บายพาส `Checks.Disabled`** เหมือนที่ `-Only`
  ทำอยู่แล้ว (ไม่งั้นต้องแก้ `Checks.Disabled` ก่อนถึงจะใช้ `-Group ArcGIS` ได้ ซึ่งขัดจุดประสงค์)
- `-Group Server` ยังเคารพ `Checks.Disabled` ตามปกติ (`WU`/`CONN` ยังปิดอยู่) แค่ตัด `AGS*` ออก
  เพิ่มเติม
- คำสั่งเดิมที่เอกสารไว้ก่อนหน้า (`-Only AGS,AGSSVC,AGSUSAGE` แบบไม่มี `-Group`) **ยังใช้ได้เหมือนเดิม
  ทุกประการ** ไม่กระทบ scheduled task หรือสคริปต์เก่าที่เคยเขียนไว้

**เมนู (`Show-PMMenu.ps1`)** เพิ่มหน้าจอ "What do you want to check?" เป็นหน้าแรก เลือก `[1] Server`
ไปหน้าเดิม (เก็บตัวอย่าง CPU/หน่วยความจำ) เพิ่มปุ่ม `[B] Back` ให้ย้อนกลับมาหน้านี้ได้ เลือก
`[2] ArcGIS Server` ตรวจทันทีไม่มีขั้นตอนเก็บตัวอย่าง (ไม่เกี่ยวกับ CPU/หน่วยความจำ) ถ้ายังไม่ได้ตั้งค่า
การเชื่อมต่อจะพาไปหน้าตั้งค่า (`Show-PMArcGISMenu`) ให้ก่อนอัตโนมัติ

**ตัดสินใจไม่ recurse กลับเข้าไฟล์ตัวเองตอน `[B]`** ลองแบบ `& $MyInvocation.MyCommand.Path ...` ก่อน
แล้วเปลี่ยนใจ เพราะกองเฟรมสคริปต์ซ้อนกันไปเรื่อย ๆ โดยไม่จำเป็น ใช้ลูป `while (-not $ready)` ครอบ
ทั้งสองหน้าจอแทน ตัวแปร `$backToModeScreen` ในหน้าจอที่สองบอกลูปนอกให้วนกลับไปหน้าจอแรกแทนที่จะตั้ง
`$ready = $true`

ทดสอบด้วยการ pipe input ครบทุกเส้นทาง (`1>B>Q` วนกลับหน้าแรกได้จริง, `2` ตอนมีการเชื่อมต่อแล้ววิ่ง
ตรงเข้าตรวจ ArcGIS ทันทีไม่มีหน้าเก็บตัวอย่าง, `2` ตอนไม่มีการเชื่อมต่อพาไปตั้งค่าให้ก่อน — ต้อง
ทดสอบซ้ำเพราะรอบแรกจัดลำดับ input ผิดคิวเนื่องจากลืมนับพรอมต์ "Press Enter to continue" ที่แทรกอยู่)
และทดสอบ `-Group`/`-Only` ผสมทุกแบบตรงกับ `Start-PMCheck.ps1` โดยตรง (ผสมกลุ่มแล้ว throw, ระบุ
`-Group ArcGIS` แล้วบายพาส Disabled จริง, คำสั่งเดิมไม่มี `-Group` ยังรันได้เหมือนเดิม) รันจริงกับ
ไซต์ ArcGIS จริงทั้งผ่านเมนูและผ่าน `-Group ArcGIS` ตรง ๆ ได้ผลถูกต้องทั้งคู่

**เอกสารที่แก้ตาม**: `CHECKS.md`, `USAGE.md`, `Manual.html` (หัวข้อที่ 6 เปลี่ยนชื่อจาก "เมนูเลือกเวลา
เก็บข้อมูล" เป็น "เมนูเลือกสิ่งที่จะตรวจ" และเพิ่มภาพหน้าจอใหม่ ตรวจซ้ำด้วยการเปิดดูจริงในเบราว์เซอร์
ไม่ใช่แค่อ่าน diff)

### `AGSDATA` — การเชื่อมต่อฐานข้อมูลที่ลงทะเบียนไว้ (23 ก.ค. 2569) — ยืนยันกับไซต์จริงแล้ว

เขียนตามแผนเดิม (`/data/validateAllDataItems`) แล้วพบว่าใช้ไม่ได้จริงตั้งแต่สำรวจ endpoint —
**endpoint นี้คืนแค่ `{"status":"success"}` รวมทั้งไซต์ ไม่มีรายละเอียดต่อรายการเลย** ใช้แยกรายการ
เชื่อมต่อได้/ไม่ได้ไม่ได้ นี่เป็นครั้งที่สี่แล้วที่ schema จริงต่างจากที่เดาหรือจากเอกสาร Esri
(ต่อจาก `/machines`, service state, และ `/usagereports`) — ยืนยันว่ากฎ **"สำรวจ API จริงก่อนเขียนโค้ด
เสมอ" ยังคุ้มค่าทุกครั้ง**

**เปลี่ยนมาใช้สอง endpoint แทน**: `POST /data/findItems` (`ancestorPath=/`, `types=egdb,folder,
bigDataFileShare,nosql,cloudStore,rasterStore`) เพื่อดึงรายการที่ลงทะเบียนไว้ทั้งหมดในคำสั่งเดียว
(recursive อยู่แล้ว ไม่ต้องไล่ทีละหมวดใน `rootItems`) แล้ว `POST /data/validateDataItem` ทีละรายการ
ซึ่งตอบ `{"status":"success"}` หรือ `{"status":"error","messages":[...]}` ต่อรายการจริง ๆ

**ข้อควรระวังด้านความปลอดภัยที่สำคัญที่สุดของหัวข้อนี้** — `findItems` คืนฟิลด์ `info.connectionString`
ของ enterprise geodatabase มาด้วย ซึ่งมี **รหัสผ่านที่เข้ารหัสไว้ (`ENCRYPTED_PASSWORD`)** รวมถึงชื่อ
เซิร์ฟเวอร์ ชื่อฐานข้อมูล และชื่อบัญชีผู้ใช้ครบ ก้อนนี้ใช้แค่ในหน่วยความจำเพื่อสร้าง request body ของ
`validateDataItem` รายการเดียวกันเท่านั้น **ไม่เคยถูกใส่ลง `$rows`/`$findings`/`-Raw`** จึงไม่มีทาง
หลุดไปอยู่ใน `PM-Data.json` หรือ `PM-Report.html` เลย ทดสอบยืนยันแล้วทั้งกับไซต์จริง (grep หา
`ENCRYPTED_PASSWORD`/`SERVER=`/`connectionString` ในทั้งสองไฟล์ผลลัพธ์ ไม่เจอ) และกับ mock ที่ตั้งใจ
ใส่ connection string ปลอมไว้ในทุกสถานการณ์ทดสอบ (grep หาในผลลัพธ์ที่ serialize แล้วของทุกเคส
ไม่เจอเช่นกัน)

**บั๊กที่เจอระหว่างทาง (ไม่เกี่ยวกับ ArcGIS)** — `Get-PMWord -Key (if (...) {...} else {...})` ใช้ไม่ได้
เพราะ `if` เป็น statement ไม่ใช่ expression วงเล็บเฉย ๆ `(...)` ใช้ครอบ statement ไม่ได้ ต้องเป็น
subexpression `$(...)` เท่านั้น (`if` ไม่ถูกมองว่าเป็นคำสั่งที่รู้จัก กลายเป็น error
"The term 'if' is not recognized...") พังตั้งแต่รันจริงครั้งแรก แก้โดยแยกเป็น if/else ธรรมดาสองบรรทัด
ก่อนเรียก `Get-PMWord`

ทดสอบครบ: ไซต์จริง (4 รายการ ผ่านหมด) และ mock 4 สถานการณ์ (ปกติ, มีรายการเชื่อมต่อไม่ได้ 1 รายการ,
อ่านรายการทั้งหมดไม่ได้, ไม่มีรายการลงทะเบียนเลย) ทุกเคสตรวจ leak เพิ่มด้วย ไม่เจอสักเคส

### `AGSLOG` — บันทึกเหตุการณ์ระดับ SEVERE/WARNING (23 ก.ค. 2569) — ยืนยันกับไซต์จริงแล้ว

หัวข้อ ArcGIS สุดท้ายที่วางแผนไว้ ผู้ใช้ขอให้จำกัดจำนวนไว้ที่ 100 รายการต่อระดับเพื่อความเร็ว
(`ArcGIS.LogPageSize`) สำรวจ `/admin/logs/query` กับไซต์จริงก่อนเขียนโค้ดตามกฎเดิม พบสามจุดสำคัญ

1. **`filter` เป็นพารามิเตอร์บังคับ** เหมือน `AGSUSAGE` — เรียกไม่ส่ง `filter` ได้ HTTP 500
   `Invalid filter value. / missing query parameters` ค่าที่ใช้ได้คือ
   `{"codes":[],"processIds":[],"users":[],"server":"*"}`

2. **`level` เป็นเกณฑ์ขั้นต่ำ ไม่ใช่ตัวกรองตรงตัว** — `level=WARNING` คืนทั้ง WARNING และ SEVERE
   ปนกัน เรียงล่าสุดก่อน **พบปัญหาจริงกับไซต์ทดสอบ**: มี WARNING ตัวเดียว ("service not found" จาก
   ไฟล์ static data ที่ถูกลบไปแล้ว) เกิดถี่มากจนเต็มหน้าแรกของผลลัพธ์ 100 รายการทั้งหมด ถ้า query
   รวมกันครั้งเดียว **SEVERE ตัวจริงที่เกิดก่อนหน้าจะถูกบังไม่ให้โผล่มาเลย** แก้โดยแยก query
   `level=SEVERE` กับ `level=WARNING` เป็นสองครั้ง แล้วกรองผลแต่ละชุดด้วย `type` ของตัวเองอีกชั้น
   ไม่ให้นับซ้ำ — พิสูจน์คุ้มค่าทันทีที่รันกับไซต์จริง: เจอ SEVERE จริง 6 รายการ (บริการ
   `AGMCoreTools/Envelop.GPServer` หาไม่เจอ) ที่จะไม่มีทางเห็นถ้า query รวมกันครั้งเดียว

3. **error ของ endpoint นี้มาเป็น `{"status":"error","messages":[...]}` ไม่ใช่ `{"error":{...}}`**
   เหมือน `AGSDATA`/`AGSUSAGE` — `Invoke-PMArcGISAdmin` ไม่ throw ให้อัตโนมัติกับรูปแบบนี้
   **บั๊กที่เจอจริงจากการทดสอบด้วย mock** (ไม่ใช่ไซต์จริง เพราะไซต์จริงให้สิทธิ์อ่าน log ครบ):
   โค้ดรอบแรกพึ่ง `try/catch` เฉย ๆ แล้วอ่าน `$resp.logMessages` ต่อ ได้ `$null` เงียบ ๆ
   กลายเป็น "ไม่พบปัญหา" ทั้งที่ query ล้มเหลวจริงจากสิทธิ์ไม่พอ — แก้โดยตรวจ `$resp.status` เอง
   อีกชั้น และถ้าอ่านไม่ได้ทั้งสองระดับ จะขึ้น `WARN` "อ่านบันทึกไม่ได้" แทนที่จะขึ้น `OK` เงียบ ๆ
   (หลักเดียวกับที่ใช้มาตลอด: **สถานะที่อ่านไม่ได้ ต้องรายงานว่าไม่รู้ ไม่ใช่รายงานว่าปกติ**)

ทดสอบครบ: ไซต์จริง (`-Group ArcGIS` รวม 5 หัวข้อ รันได้หมด, AGSLOG ขึ้น `CRIT` ถูกต้องจาก SEVERE
จริงที่เจอ) และ mock 2 สถานการณ์ (ไม่มีบันทึกเลย, อ่านบันทึกไม่ได้ทั้งสองระดับ)

**ครบทั้ง 5 หัวข้อ ArcGIS ที่วางแผนไว้แล้ว** (`AGS`, `AGSSVC`, `AGSUSAGE`, `AGSDATA`, `AGSLOG`)
ไม่มีหัวข้อ ArcGIS ที่ออกแบบไว้ค้างอยู่อีก

## ค้างมาจากรอบก่อน — ยังไม่ได้ยืนยัน

> ยกมาจากบันทึกวันที่ 21 ก.ค. **ยังไม่ได้ทำ** เพราะยังไม่มีโอกาสรันบนเครื่องที่มีอาการ

> ยืนยันได้แค่ว่าเร็วขึ้น (3,048 มิลลิวินาที เหลือ 194) และข้อมูลถูกต้องครบ

## `Build-PMSingle.ps1` — ตัวรวมเป็นไฟล์เดียว

**22 ก.ค. 2569** เพิ่มตัว build ที่รวมทั้งชุดเป็นไฟล์เดียวแตกตัวเองได้ เพื่อให้ส่งไปเครื่องแม่ข่ายง่ายขึ้น
รายละเอียดการใช้งานอยู่ใน `DISTRIBUTION.md` หัวข้อ *แจกแบบไฟล์เดียว* ตรงนี้เก็บเฉพาะเหตุผลเชิงออกแบบ

**ทำไมเลือกวิธี "ห่อ" ไม่ใช่ "รวมโค้ด"** — ไฟล์ที่ได้บรรจุ zip ของซอร์สทั้งชุดเป็น base64
ตอนรันจะแตกลงโฟลเดอร์ชั่วคราวแล้วเรียก `Show-PMMenu.ps1` / `Start-PMCheck.ps1` ตัวเดิม
ไม่มีการแปลงหรือต่อโค้ดใด ๆ เลย ทางเลือกอีกทางคือรวมทุกไฟล์เป็นสคริปต์เดียวจริง ๆ ซึ่งต้อง
ตัด `param()` ออก เปลี่ยนวิธี discover หัวข้อ และฝัง JSON แทนการอ่านไฟล์ — แปลว่าทุกครั้งที่แก้ซอร์ส
ตัว build อาจพังตาม **แบบห่อนี้เพิ่มหัวข้อตรวจใหม่แล้ว build ซ้ำได้เลย ไม่ต้องแตะตัว build**

**ทำไม `.cmd` ถึงเป็นค่าเริ่มต้น** — cmd.exe อ่านไฟล์ batch ทีละบรรทัดตอนรัน
บรรทัดที่อยู่หลัง `exit /b` จึงไม่ถูก parse เลย ทำให้เอา PowerShell กับ base64 ไปต่อท้ายไฟล์ `.cmd` ได้
ส่วน batch ข้างบนอ่านไฟล์ตัวเองกลับเข้ามาแล้วเรียกเฉพาะท่อนหลัง marker
ได้ผลคือไฟล์เดียวที่ดับเบิลคลิกได้ ขอสิทธิ์ผู้ดูแลระบบเองได้ และไม่ติด execution policy
(`.ps1` คลิกขวาสั่ง *Run as administrator* ไม่ได้ ซึ่งเป็นเหตุผลหลักที่ไม่เลือก)

### สองจุดที่พลาดตอนทำ ทั้งคู่เกี่ยวกับการส่ง argument ต่อ

1. **splat ด้วย array จะ bind แบบ positional ไม่ใช่ named**
   `$a = @('-OutputRoot', $path); & .\Start-PMCheck.ps1 @a` ไม่ได้แปลว่าส่ง `-OutputRoot`
   แต่แปลว่าส่งสตริง `'-OutputRoot'` เป็นค่าพารามิเตอร์ตัวแรก และ `$path` เป็นตัวที่สอง
   ผลคือ `$ConfigDir` กลายเป็น `PM-Output\` แล้วพังตอนหา `settings.json`
   **มีแต่ hashtable เท่านั้นที่ splat แบบ named** ตัว bootstrap จึงแปลง token เป็น hashtable ก่อน

2. **`powershell -File` ส่ง argument เป็นสตริงล้วนเสมอ**
   `-Only DISK,CERT` ผ่าน `-File` จะมาเป็นสตริงเดียว `'DISK,CERT'` ไม่ใช่ array
   (ต่างจาก `-Command` ที่ parse ให้เป็น array) พอ bind เข้า `[string[]]$Only` เลยได้ลิสต์
   ที่มีสมาชิกเดียวชื่อ `DISK,CERT` ซึ่งไม่ตรงกับ id ใด สุดท้ายขึ้นว่า "No checks were selected"
   ตัว bootstrap จึงแตกค่าของ `-Only` กับ `-Skip` ด้วยเครื่องหมายจุลภาคเอง

   ต่อจากบทเรียน `Run-PM.cmd` ข้างบนโดยตรง — **สองเส้นทางนี้ไม่เหมือนกัน ต้องทดสอบทั้งคู่**

### ทดสอบตัวไฟล์เดียวโดยไม่ต้องเจอหน้าต่างขอสิทธิ์

ท่อน batch จะเรียก `Start-Process -Verb RunAs` ถ้ายังไม่ได้สิทธิ์ผู้ดูแลระบบ
ซึ่งทำให้ทดสอบอัตโนมัติไม่ได้ ให้เรียกท่อน PowerShell ตรง ๆ แบบเดียวกับที่ label `:elevated` ทำ

```powershell
$env:PM_SELF = 'D:\PMtools\Dist\PMtools-1.7.0.cmd'
$boot = '$t=[IO.File]::ReadAllText($env:PM_SELF); $m=''#PM''+''TOOLS_PAYLOAD#''; & ([scriptblock]::Create($t.Substring($t.IndexOf($m))))'
powershell -NoProfile -ExecutionPolicy Bypass -Command "$boot -Only DISK"
```

สิ่งที่ควรตรวจหลัง build ทุกครั้ง: ภาษาไทยในรายงานยังครบ, โฟลเดอร์ `<ชื่อเครื่อง>_<วันเวลา>\`
ออกข้างตัวไฟล์โดยตรง (ไม่มีชั้น `Output\` คั่น — ดูหัวข้อถัดไปว่าทำไมเปลี่ยนจาก `PM-Output\` เดิม),
โฟลเดอร์ `%TEMP%\PMtools-*` ถูกลบหมด, และ `-ExtractConfig` แล้วลบ `i18n.json` ทิ้งยังรันได้
(ทดสอบว่า override แบบไม่ครบไฟล์ยังไม่พัง)

**เปลี่ยนจาก `PM-Output\` เป็นออกข้างไฟล์ตรง ๆ ทีหลัง** ตามที่ผู้ใช้ขอ — เดิม `Invoke-PMSingleFile`
สร้างโฟลเดอร์ `PM-Output\` มาคั่นไว้ก่อนเสมอ ตอนนี้ตรวจว่าโฟลเดอร์ที่ตัวไฟล์อยู่เขียนได้จริงหรือไม่
ด้วยการสร้างไฟล์ทดสอบเล็ก ๆ แล้วลบทิ้ง (ไม่ใช่แค่เช็คว่าโฟลเดอร์มีอยู่ เพราะโฟลเดอร์ที่มีอยู่แล้ว
เขียนไม่ได้ก็ยังผ่าน `Test-Path`) ถ้าเขียนไม่ได้ค่อย fallback ไป `%TEMP%\PM-Output\`

## `Export-PMDocxReport.ps1` — ส่งออกเป็น Word

**22 ก.ค. 2569** เพิ่มเครื่องมือแยกที่แปลง `PM-Data.json` เป็นเอกสาร Word สองภาษา
ผ่าน COM automation (`New-Object -ComObject Word.Application`) รายละเอียดการใช้งาน
อยู่ใน `DISTRIBUTION.md` หัวข้อ *ส่งออกเป็น Word (.docx)* และ Manual.html หัวข้อที่ 5
ตรงนี้เก็บเฉพาะเหตุผลเชิงออกแบบและกับดักที่เจอจริงระหว่างทำ — **ทุกข้อล่างนี้พิสูจน์ด้วยการ
รัน Word จริงแล้วแปลงผลลัพธ์เป็น PDF ดูภาพ ไม่ใช่เดาจากเอกสาร COM**

**ทำไมแยกเป็นเครื่องมือคนละตัว รันคนละเครื่อง** — ตัวตรวจต้องอ่านค่าอย่างเดียวและรันบนเครื่อง
แม่ข่ายโดยไม่พึ่งโปรแกรมเสริม (หลักการเดิมของทั้งโปรเจกต์) แต่ Word COM ต้องมี Microsoft Word
ติดตั้งอยู่ ซึ่งไม่ควรบังคับให้เครื่องแม่ข่ายต้องมี จึงแยกเป็น `Export-PMDocxReport.ps1` ที่รันบน
เครื่องของผู้ดูแลระบบเองแทน อ่านแค่ไฟล์ `PM-Data.json` ไฟล์เดียว ไม่ต้องแตะเครื่องแม่ข่ายอีกครั้ง
`Lib\Report.Docx.ps1` จึงถูกกันออกจาก `Build-PMSingle.ps1` โดยตั้งใจ (ดู `$bundleExclude`)
เพราะไม่มีประโยชน์อะไรบนเครื่องแม่ข่ายที่ไม่มี Word

**`PM-Data.json` ไม่มีข้อมูลหน่วยงาน** — ตั้งใจตัด `Config` ออกตอนเขียน JSON (ดู `Start-PMCheck.
ps1`) ตัวแปลง Word จึงอ่านชื่อหน่วยงาน/ส่วนงาน/ผู้ตรวจสอบจาก `Config\settings.json` ของ**เครื่องที่
รันตัวแปลงเอง** ไม่ใช่เครื่องที่ถูกตรวจ — ถ้าค่าสองเครื่องไม่ตรงกัน หัวรายงาน Word กับ HTML จะไม่
เหมือนกัน เป็นเรื่องที่ต้องรู้ไว้ ไม่ใช่บั๊ก

**สามกับดักที่เจอจริงจาก Word COM automation** (ไม่มีในเอกสารทางการ ต้องรันดูเอง)

1. **Thai เป็น Complex Script ในโมเดลฟอนต์ของ Word** เหมือน Arabic/Hebrew ทั้งที่ไม่ใช่ภาษาเขียน
   ขวาไปซ้าย ตั้งฟอนต์ผ่าน `Font.Name`/`NameAscii` อย่างเดียวไม่พอ — ตัวอักษรละตินที่ปนอยู่ในบรรทัด
   เดียวกับไทยจะหลุดไปใช้ฟอนต์ธีมของสไตล์แทน ต้องตั้งครบ `NameAscii` + `NameOther` + `NameBi`
   ส่วน `NameFarEast` ข้ามได้ (และต้องข้าม — จะ throw ถ้าเครื่องไม่ได้ลง East Asian language pack
   ซึ่ง Thai ไม่ต้องพึ่งอยู่แล้ว)

2. **`Range.End` ของเซลล์ตารางรวมเครื่องหมายจบเซลล์ (end-of-cell mark) ไว้ด้วย** จะแทรกข้อความ
   บรรทัดที่สองต่อในเซลล์เดิม ต้อง `MoveEnd(1, -1)` ถอยออกมาก่อนเสมอ ข้ามขั้นนี้แล้วตารางจะเพี้ยน
   แบบไม่ throw ให้เห็น (แถวเกิน หรือข้อความรั่วไปเซลล์ถัดไป) — เจอจากการรัน Word จริงแล้วเปิดไฟล์
   กลับมานับจำนวนพารากราฟต่อเซลล์ ไม่ใช่จากการอ่านโค้ด

3. **ห้ามข้ามการแทรกพารากราฟเพื่อประหยัดช่องว่าง** เวอร์ชันแรกใช้พารามิเตอร์ `-NoLeadingBreak`
   เพื่อลดช่องไฟระหว่างบรรทัดไทย/อังกฤษที่ต่อกัน แต่การข้าม `InsertParagraphAfter()` ทำให้ข้อความ
   สองภาษาต่อกันในบรรทัดเดียวไม่มีตัวคั่นเลย (เห็นชัดตอนแปลงเป็น PDF ดู: "...14 หัวข้อThe assessment
   ...") ที่จริง Word ใส่ระยะห่างหลังพารากราฟ (`SpaceAfter`) 8pt ให้เป็นค่าเริ่มต้นอยู่แล้ว
   วิธีแก้ที่ถูกคือขึ้นพารากราฟใหม่เสมอ แต่ตั้ง `ParagraphFormat.SpaceBefore`/`SpaceAfter` เป็น 0
   เฉพาะบรรทัดที่ต้องการให้ชิดกัน (`-Tight` ใน `Add-PMWordParagraph`)

**เรื่องอื่นที่ปรับหลังเห็นผลจริง**: ตารางที่ยาวเกินหนึ่งหน้าเดิมปล่อยให้ Word ตัดแถวกลางแถวได้ตาม
ค่าเริ่มต้น (เห็นแถว "Normal" ของไดรฟ์ C: หลุดไปอยู่คนละหน้ากับส่วนที่เหลือของแถว) แก้ด้วย
`$tbl.Rows.AllowBreakAcrossPages = 0` ทุกตาราง และตั้ง `HeadingFormat = -1` ให้แถวหัวตาราง
ของตารางภาพรวมกับตารางรายละเอียดซ้ำทุกหน้าที่ตารางไหลข้ามไป

**ยังไม่ทำในรุ่นนี้**: กราฟแนวโน้ม CPU/หน่วยความจำ (หัวข้อ TREND) ไม่ได้วาดใน Word ตั้งใจข้ามไว้
เพราะ Word Chart ผ่าน COM (`InlineShapes.AddChart2`) ต้องยุ่งกับ embedded workbook ผ่าน COM
ซึ่งเสี่ยงค้างและซับซ้อนเกินความคุ้มค่าของรุ่นแรก ตารางตัวเลข (ต่ำสุด/เฉลี่ย/เปอร์เซ็นไทล์ที่ 95/สูงสุด)
ยังอยู่ครบ แค่กราฟเส้นแบบโต้ตอบได้ไม่มี — มีข้อความกำกับในเอกสาร Word บอกให้ไปดูที่ HTML แทน

**กับดักใหม่ที่เจอ — Nullable[double] ผ่าน COM ทำให้ `Font.Size` โยน `InvalidCastException`**
เวอร์ชันแรกของ `Add-PMWordParagraph` ประกาศ `[Nullable[int]]$Style` / `[Nullable[double]]$Size` /
`[Nullable[int]]$Color` เพื่อให้พารามิเตอร์ที่ไม่ได้ระบุมีค่าเป็น `$null` ทดสอบแบบแยกส่วน (เอกสารเปล่า
ไม่มีตาราง) ผ่านทุกครั้ง แต่พอรันกับข้อมูลจริง (มีตารางมาสท์เฮด/meta grid/tiles แทรกไว้ก่อนแล้ว) บรรทัด
`$r.Font.Size = $Size` โยน `Specified cast is not valid` ทันที — **ไล่ด้วยการใส่ `Write-Host` คั่นทุก
บรรทัดในฟังก์ชันจริงแล้วรันกับข้อมูลจริง ไม่ใช่จากการอ่านโค้ดหรือเทสต์แบบง่าย** เพราะ repro แบบง่าย
(เอกสารเปล่า, ตั้งค่าเดียวกันทุกอย่าง) ไม่ล้มเลย ต้องมีบริบทเอกสารจริงถึงจะเจอ

แก้โดยเลิกใช้ `Nullable<T>` ทั้งสามพารามิเตอร์ เปลี่ยนเป็น `[int]$Style = 0` / `[double]$Size = 0` /
`[int]$Color = 0` แล้วเช็ค `-ne 0` แทน `$null -ne` — ปลอดภัยเพราะไม่มีค่าจริงของทั้งสามตัวที่ควรเป็น 0
พอดี (wdBuiltinStyle ไม่มีค่า 0, ไม่มีฟอนต์ขนาด 0, และสีดำล้วนคือค่าเริ่มต้นของ Word อยู่แล้วไม่มี
จุดไหนต้องสั่ง 0,0,0 ตรง ๆ) **ถ้าจะส่งค่าตัวเลขเข้า Word COM property ในอนาคต หลีกเลี่ยง `Nullable<T>`
ไว้ก่อน ใช้ sentinel value แทน**

### บั๊กเดิมที่เจอระหว่างทาง — `continue` ใน `switch` ไม่ได้วนลูป

เมนูเดิมเขียนแบบนี้ และ**ผิดมาตั้งแต่ต้น**

```powershell
while ($true) {
    switch ($choice) {
        '1'     { ...; break }
        default { 'invalid'; continue }   # ตั้งใจให้วาดเมนูใหม่
    }
    break                                  # <- ตกมาโดนบรรทัดนี้
}
```

PowerShell ผูก `break`/`continue` เข้ากับ **switch** ไม่ใช่ `while` ที่ครอบอยู่ — `continue`
จึงแค่ออกจาก switch แล้วตกลงมาโดน `break` ซึ่งออกจาก while **ผลคือกดปุ่มผิดในเมนู
แล้วเครื่องมือรันการตรวจสอบเต็มรูปแบบทันทีแทนที่จะวาดเมนูใหม่** และการยกเลิกตอนถาม
custom duration ก็รันด้วย `-Minutes 0` เงียบ ๆ เหมือนกัน

พิสูจน์ด้วยการรัน scriptblock จริงดูพฤติกรรม ไม่ใช่อ่านโค้ดเดา แก้โดยเลิกพึ่ง break/continue
เปลี่ยนเป็น `while (-not $chosen)` แล้วให้แต่ละ case ตั้ง `$chosen` เอง

### ข้อจำกัดตอนทดสอบ — `Read-Host -AsSecureString` ไม่รับ stdin ที่ pipe เข้าไป

ยืนยันด้วยการทดสอบแล้ว: มันอ่านจาก console host โดยตรง ทำให้**ทดสอบขั้นตอนกรอกรหัสผ่าน
แบบอัตโนมัติไม่ได้เลย** — process จะค้างรอคีย์บอร์ดจนหมดเวลา (ครั้งแรกที่เจอคือ pipe ค้างเกิน
120 วินาที ตอนแรกนึกว่าเน็ตเวิร์กค้าง แต่ทดสอบแล้ว request จริง fail ใน 0.1 วินาที)

เวลาทดสอบเมนูให้ pipe ได้ถึงแค่ก่อนถามรหัสผ่าน ส่วนที่เหลือต้องกดมือ และใช้ `Start-Transcript`
ถ้าอยากเห็นหน้าจอที่ `Clear-Host` ลบไปแล้ว

### ทดสอบด้วย mock ArcGIS Server

จุดสำคัญของ mock คือมันเลียนแบบพฤติกรรมที่ทำให้โค้ดนี้ต้องเขียนแบบระวัง: **ตอบ HTTP 200
พร้อม `{"error":{...}}` ใน body** เวลารหัสผ่านผิด ไม่ใช่ตอบ HTTP 401 — ถ้า mock ตอบ 401
จะไม่ได้ทดสอบโค้ดส่วนที่สำคัญที่สุดเลย

ผลที่ได้ครบทุกสถานการณ์

| สถานการณ์ | ผลที่ได้ | ตรวจอะไร |
|---|---|---|
| ปกติ 2 เครื่อง | `OK` | token + `/info` + `/machines` ครบเส้นทาง |
| เครื่องหนึ่งหยุด | `CRIT` + finding ระบุชื่อเครื่อง | `realTimeState` ต่างจาก `configuredState` |
| บัญชีไม่มีสิทธิ์อ่าน machines | `WARN` (ไม่ใช่พัง) | เชื่อมต่อได้แต่อ่านบางอย่างไม่ได้ |
| รหัสผ่านผิด | `ERROR` + ข้อความจากเซิร์ฟเวอร์ | จับ error ที่ซ่อนใน body ของ HTTP 200 |
| ยังไม่ตั้งค่าการเชื่อมต่อ | `ERROR` + บอกวิธีแก้ | ไม่ทำให้ทั้ง run พัง |

ตรวจเพิ่มเติมแล้วว่า **รหัสผ่านและ token ไม่ปรากฏใน `PM-Data.json` หรือ `PM-Report.html`**
และรายงาน HTML ไม่มีอักขระ U+FFFD (ภาษาไทย 714 ตัวอักษร ครบถ้วน)

### บทเรียนสำคัญ — mock ที่เดา schema เอง ทำให้เกิด false positive บน production

**22 ก.ค. 2569** ทดสอบกับไซต์จริงครั้งแรก (ArcGIS Server **11.5.0** ผ่าน Web Adaptor บน HTTPS)
ผลคือ **รายงานว่าเครื่องทั้ง 2 เครื่อง "ไม่ทำงาน" ระดับวิกฤต ทั้งที่ทั้งคู่ `STARTED` ปกติดี**

สาเหตุ: `/admin/machines` ของจริง**ไม่คืนสถานะเลย** คืนแค่ 4 ฟิลด์

```
machineName, adminURL, synchronize, underMaintenance
```

สถานะอยู่ที่ **`/admin/machines/<ชื่อเครื่อง>/status`** ต้องเรียกแยกทีละเครื่อง
(คืน `configuredState` + `realTimeState`)

แต่ mock ที่เขียนไว้**ใส่ `configuredState`/`realTimeState` ลงใน `/machines` เอง** ตามที่เดาว่าน่าจะมี
โค้ดจึงอ่านได้ค่าว่างบนของจริง แล้วตีความว่า "ไม่ใช่ STARTED = หยุดทำงาน" → CRIT

**สองบทเรียน**

1. **mock ที่ schema ไม่ตรงของจริง อันตรายกว่าไม่มี mock** เพราะมันให้ความมั่นใจปลอม ๆ
   ตอนนี้แก้ mock ให้ตรงของจริงแล้ว พร้อมคอมเมนต์กำกับไว้ว่าห้ามใส่ฟิลด์ที่ API จริงไม่มี
2. **สถานะที่อ่านไม่ได้ ต้องไม่ถูกนับเป็น "พัง"** เดิมโค้ดรวม "ว่าง" เข้ากับ "หยุดทำงาน"
   ตอนนี้แยกเป็น UNKNOWN → `WARN` พร้อมบอกเหตุผล **เครื่องมือเฝ้าระวังที่แจ้งเตือนวิกฤตผิด
   บนระบบที่ปกติดี แย่กว่าเครื่องมือที่ยอมรับว่ามันไม่รู้**

หลังแก้ รันกับไซต์จริงได้ `OK` ตรงกับความจริง และ mock ครบ 4 สถานการณ์
(`healthy` → OK, `stopped-machine` → CRIT, `no-state` → WARN, `no-machine-access` → WARN)

เจอตอนลองรัน `AGS` รวมกับหัวข้ออื่นด้วย `powershell -File ... -Only SYSTEM,DISK,CERT,AGS`
แล้วได้ **"No checks were selected"** ทั้งที่ทุก id มีอยู่จริงในรายการที่ error พิมพ์ออกมาเอง

สาเหตุเดิมซ้ำรอบที่สาม: `-File` ส่ง argument เป็นสตริงล้วน `-Only SYSTEM,DISK` จึงมาเป็น
**1 สมาชิก** `"SYSTEM,DISK"` ไม่ใช่ 3 สมาชิก (พิสูจน์แล้วด้วยสคริปต์ทดสอบเทียบสองวิธีเรียก)

รอบก่อน ๆ แก้เฉพาะที่ bootstrap ของไฟล์เดียว แต่ **`Start-PMCheck.ps1` ที่ถูกเรียกตรง ๆ ยังพลาดอยู่**
ซึ่งเป็นเส้นทางที่ Task Scheduler และไฟล์ batch ใช้ ตอนนี้เพิ่ม `Expand-PMIdList` แตกจุลภาคเอง
ทั้ง `-Only` และ `-Skip` — ปลอดภัยเพราะ id ของหัวข้อไม่มีจุลภาคอยู่แล้ว

ทดสอบครบ 4 รูปแบบ: `-File` + จุลภาค, `-Skip` + จุลภาค, ค่าเดียว, และ `&` แบบ array เดิม
รวมถึงการรันปกติที่ยังเคารพ `Checks.Disabled`

### `AGSSVC` — บทเรียนเดิมซ้ำอีกครั้ง แต่คราวนี้ดักไว้ทัน

สำรวจไซต์จริงก่อนเขียนโค้ด พบว่ามี **139 บริการ และ 16 รายการที่ `realTimeState` ไม่ใช่ `STARTED`**
ถ้าเขียนตามสัญชาตญาณว่า "ไม่ทำงาน = วิกฤต" ไซต์ที่แข็งแรงดีจะขึ้นเตือนผิด 16 รายการ

แต่ดูให้ลึกอีกชั้น **ทั้ง 16 รายการมี `configuredState = STOPPED` ด้วย** คือถูกตั้งใจปิด
(ส่วนใหญ่คือบริการในโฟลเดอร์ `System` และ `Utilities` ที่ ArcGIS ติดตั้งมาแล้วปิดไว้)

**เกณฑ์ที่ถูกคือเทียบ configured กับ realTime ไม่ใช่ดูแค่ realTime**

| configured | realTime | ผล |
|---|---|---|
| STARTED | STARTED | OK |
| STARTED | STOPPED | **CRIT** — พังจริง |
| STOPPED | STOPPED | INFO — ตั้งใจปิด |
| STOPPED | STARTED | WARN |

เป็นบทเรียนเดียวกับเรื่องสถานะเครื่องเป๊ะ ๆ (**สถานะที่ตั้งใจ ไม่ใช่ความผิดปกติ**) ต่างกันแค่คราวนี้
สำรวจก่อนเขียน เลยดักได้ก่อนที่มันจะกลายเป็น false positive บน production

ทดสอบครบ 5 กรณีด้วย mock (`STARTED/STARTED`, `STARTED/STOPPED`, `STOPPED/STOPPED`,
`STOPPED/STARTED`, และ busy=max) ทุกกรณีจำแนกถูก

### เรื่องจำนวนคำขอกับเวลาตอบสนอง — สิ่งที่ได้และไม่ได้

`/services/{folder}/report` ให้ `transactions` (คำขอสะสม) และ `totalBusyTime` (มิลลิวินาที)
หารกันได้ **เวลาประมวลผลเฉลี่ยต่อคำขอ** ซึ่งเป็น GET ล้วน ไม่ต้องเขียนอะไร

ค่าที่ได้จริงจากไซต์ทดสอบมีตั้งแต่บริการที่ใช้หนัก (คำขอสะสมกว่า 375,000 ครั้ง เฉลี่ย ~330 ms)
ไปจนถึงเครื่องมือประมวลผลที่ถูกเรียกไม่บ่อยแต่ใช้เวลานาน (คำขอหลักสิบ เฉลี่ยกว่า 2,300 ms)
ซึ่งเป็นสองรูปแบบภาระงานที่ต่างกันชัดเจนและเห็นได้จากตารางเดียว

**ข้อควรระวัง**: `totalBusyTime` คือเวลาที่อินสแตนซ์ถูกครอบครอง **ไม่ใช่เวลาตอบสนองปลายทาง**
ไม่รวมเวลารอคิวและเวลาบนเครือข่าย จึงตั้งชื่อคอลัมน์ว่า "เวลาประมวลผลเฉลี่ย" ไม่ใช่ "เวลาตอบสนอง"

**เรื่องที่ยังไม่ได้ทำและน่าสนใจ** — ไซต์ทดสอบ**มี usage report อยู่แล้ว** เพราะ ArcGIS Server Manager
สร้างไว้ตอนมีคนเปิดหน้าสถิติ (`managerReport: true`) และ `usagereports/settings` แสดง `enabled: True`

```
Total requests for the last 7 days      -> RequestCount        7 วัน 112,811 คำขอ
Max response times for the last 7 days  -> RequestMaxResponseTime  สูงสุด 561.1
Timed-out requests for the last 7 days  -> RequestsTimedOut    0
```

**อ่านรายงานที่มีอยู่แล้วเป็น GET ล้วน ไม่ผิดกฎ** — ที่ผิดกฎคือการ *สร้าง* รายงาน
ถ้าจะทำหัวข้อ `AGSUSAGE` ต่อ ต้องออกแบบให้ **อ่านเฉพาะรายงานที่มีอยู่ ไม่สร้างใหม่เด็ดขาด**
และถ้าไซต์ไหนไม่เคยเปิด Manager จะไม่มีรายงานเลย ต้องลดระดับลงอย่างสุภาพ ไม่ใช่ error

### `AGSUSAGE` — รายงานการใช้งาน (23 ก.ค. 2569) — **ยืนยันกับไซต์จริงแล้ว หลังพังรอบแรกตามคาด**

เพิ่ม `Checks\A2-ArcGISUsage.ps1` ตามที่ออกแบบไว้ข้างบน โดยรอบแรกเขียนจาก schema ที่จำมาจาก
เอกสาร Esri เท่านั้น (ปิดเป็นค่าเริ่มต้น ทดสอบผ่าน mock ที่เขียน schema เองครบ 6 สถานการณ์ commit
ไปแล้ว) — ตามคาด พอสั่ง `.\Start-PMCheck.ps1 -Only AGSUSAGE` กับไซต์จริงครั้งแรก **พังทันที**
ด้วย `Cannot index into a null array.` เป็นบทเรียนเดียวกับ `AGS`/`AGSSVC` ซ้ำเป็นครั้งที่สาม
(mock ที่เดา schema เองพิสูจน์ได้แค่ว่าโค้ดทำงานตามที่ตั้งใจ ไม่ได้พิสูจน์ว่าตรงกับของจริง)

**สำรวจ raw response จริงแล้วพบว่า schema ต่างจากที่เดาไว้สามจุดใหญ่**

1. **List endpoint คืนภายใต้คีย์ `metrics` ไม่ใช่ `usagereports`** ทำให้ `$list.usagereports`
   เป็น `$null` เสมอ — นี่คือสาเหตุจริงของ error ข้างต้น (`@($null)` ได้ array ที่มีสมาชิกเดียวเป็น
   `$null` แล้วโค้ดพยายาม index เข้า `.PSObject.Properties[...]` ของสมาชิกนั้น)

2. **หนึ่งรายงาน = หนึ่งตัวชี้วัด ไม่ใช่รายงานเดียวมีสามตัวชี้วัด** ไซต์ทดสอบมี 5 รายการใน list:
   2 รายการเป็น `metadata.temp = true` ชื่อเป็นตัวเลข timestamp ดิบ (สร้างชั่วคราวทุกครั้งที่มีคน
   เปิดหน้า Manager -> Server -> Statistics และมี `tempTimer` ให้หมดอายุเอง) กับ 3 รายการเป็น
   `metadata.temp = false` ชื่อเป็นข้อความคงที่ ("Total requests for the last 7 days" เป็นต้น)
   แต่ละรายงานมี `queries: [{resourceURIs, metrics}]` ของตัวเองครอบคลุมแค่ตัวชี้วัดเดียว
   ต้องกรองเอาเฉพาะ `temp = false` เท่านั้น (รายการ temp อาจหายไปเองระหว่างรัน และชื่อไม่คงที่)

3. **เรียก `/usagereports/<ชื่อ>/data` เฉย ๆ ไม่พอ ต้องส่งพารามิเตอร์ `filter` เสมอ** ไม่งั้นเซิร์ฟเวอร์
   ตอบ HTTP 500 `RequestUtil.getParameterIgnoreCase(...) is null` (Java NPE ฝั่งเซิร์ฟเวอร์ ไม่ใช่
   "ไม่มีรายงาน") ค่า `filter` ที่ใช้ได้คือเอา `since` และ `queries` ของรายงานนั้นเองจาก list มา
   ห่อเป็น JSON ส่งกลับไป — พิสูจน์แล้วว่า **`GET` ใช้ได้จริงเหมือนที่บันทึกไว้เดิม ไม่ต้องใช้ `POST`**
   เมื่อมี `filter` ครบ (ทดสอบเทียบทั้งสองวิธีกับไซต์จริง ได้ผลเหมือนกันทุกประการ)

**ผลพลอยได้ที่ทำให้โค้ดง่ายกว่าที่ออกแบบไว้เดิม** — แต่ละ entry ใน `report-data` ติดป้ายตัวเอง
ด้วย `metric-type` (เช่น `"metric-type": "RequestCount"`) จึงไม่ต้องจับคู่ตำแหน่งกับ
`metadata.metrics[i]` แบบที่ออกแบบไว้รอบแรกเลย — อ่าน `report-data` แบบแบน ๆ แล้วจัดกลุ่มด้วย
`metric-type` ตรง ๆ ได้เลย ทนทานกว่าเดิมด้วย (รองรับรายงานที่มีหลาย query/หลาย resourceURI
โดยไม่ต้องรู้ล่วงหน้า) ส่วน `metadata` ใน response ของ `/data` เป็น **JSON string** (ไม่ใช่ object
เหมือนใน response ของ list endpoint) แต่โค้ดไม่ต้องพึ่งมันแล้วเพราะใช้ `metric-type` แทน

แก้โค้ดตามนี้ทั้งหมดแล้ว รันกับไซต์จริงอีกครั้งได้ `OK` ตัวเลขตรงกับที่เคยยืนยันด้วยมือไว้ก่อนหน้า
(คำขอ 7 วันล่าสุด 112,195 ครั้ง เทียบกับ 112,811 ครั้งที่บันทึกไว้สัปดาห์ก่อน — ต่างกันเพราะเป็น
หน้าต่างเวลาแบบเลื่อน 7 วันล่าสุดตลอด ไม่ใช่ช่วงคงที่, เวลาตอบสนองสูงสุด 561.139 ตรงกับ 561.1 แทบ
เป๊ะ, คำขอหมดเวลา 0 เท่ากัน) ตรวจ `PM-Report.html` ที่ได้ด้วยว่าไม่มี U+FFFD และไบต์ภาษาไทยถูกต้อง
ครบ (ตรวจที่ระดับไบต์ดิบ ไม่ใช่ผ่านการพิมพ์ออก terminal ซึ่งเคยทำให้เข้าใจผิดว่าเพี้ยนระหว่างที่ debug
รอบนี้ — terminal เองที่ mis-decode ไม่ใช่ไฟล์)

mock เดิมที่ `scratchpad\mock-arcgis-usage.ps1` **เขียน schema ผิดตามรอบแรก ไม่ได้อัปเดตตามของจริง**
เพราะเป็นไฟล์ scratchpad ของ session ก่อน ถ้าจะทดสอบ `AGSUSAGE` ด้วย mock อีกต้องเขียนใหม่ให้ตรง
โครงสร้างที่ยืนยันแล้วสามข้อข้างต้น (คีย์ `metrics`, กรอง `temp=false`, ต้องส่ง `filter`, จัดกลุ่มด้วย
`metric-type`) — **หรือทดสอบกับไซต์จริงตรง ๆ เหมือนรอบนี้ก็ได้ เพราะเป็น GET ล้วนไม่มีความเสี่ยง**

### สิ่งที่ยืนยันแล้วกับไซต์จริง

- **HTTPS + TLS 1.2** ผ่าน Web Adaptor (`https://<host>/arcgis`) — `Initialize-PMArcGISTransport` ทำงานจริง
- **`generateToken`** ด้วย `client=requestip` ใช้ได้กับบัญชี `siteadmin`
- **ArcGIS Server 11.5.0** (ใหม่กว่าที่ mock จำลองไว้)
- แปลง URL, token, `/info`, `/machines`, `/machines/<name>/status` ครบเส้นทาง
- **โครงสร้าง JSON ของ `/usagereports` และ `/usagereports/<ชื่อ>/data`** (23 ก.ค.) — ดูหัวข้อ
  `AGSUSAGE` ด้านบนสำหรับรายละเอียดที่ต่างจากที่เดาไว้ตอนแรก

### ยังไม่ได้ยืนยัน

- **self-signed certificate** — ไซต์ที่ทดสอบใช้ใบรับรองจริง `Disable-PMArcGISCertificateCheck`
  จึงยังไม่เคยถูกใช้งานจริง
- **Portal-tier authentication** — ไซต์ที่ทดสอบใช้บัญชีของ ArcGIS Server เอง ถ้าไซต์ผูกกับ
  Portal for ArcGIS `generateToken` ของ Server อาจใช้ไม่ได้ ต้องขอ token จาก Portal แทน
- **ไซต์ที่มีมากกว่า 2 เครื่อง** — ตอนนี้เรียก status ทีละเครื่อง (1 + N ครั้ง) ไซต์ใหญ่ควรวัดเวลาก่อน
- **ไซต์ที่ไม่มีรายงานถาวรเลย (`temp = false` ไม่มี)** — โค้ด `AGSUSAGE` ลดระดับเป็น `INFO` ตามที่
  ออกแบบไว้ แต่ยังไม่เคยเจอไซต์จริงที่อยู่ในสภาพนั้นเพื่อยืนยัน (ไซต์ทดสอบมี Manager ถูกเปิดใช้แล้ว)

## กฎที่ห้ามผิดเวลาแก้โค้ด

สี่ข้อนี้เป็นข้อตกลงที่ทั้งระบบยึดอยู่ ถ้าผิดจะพังแบบหายาก

1. **ไฟล์ `.ps1` ต้องเป็น ASCII ล้วน ห้ามพิมพ์ภาษาไทยลงไปแม้แต่ในคอมเมนต์**
   เพราะ PowerShell 5.1 อ่าน `.ps1` เป็น ANSI ถ้าไม่มี BOM ไบต์ไทยจะกลายเป็นอักขระอื่น
   และอาจทำให้ parse พังได้ (เคยพลาดมาแล้วครั้งหนึ่งใน `Report.Html.ps1`)
   ข้อความไทยทั้งหมดอยู่ใน `Config\i18n.json` เท่านั้น

   ตรวจก่อน commit ทุกครั้ง — ต้องไม่มีผลลัพธ์ออกมา

   ```powershell
   $ansi=[System.Text.Encoding]::GetEncoding(1252)
   Get-ChildItem D:\PMtools -Recurse -Include *.ps1,*.cmd | ForEach-Object {
       $u=[System.IO.File]::ReadAllText($_.FullName,[System.Text.Encoding]::UTF8)
       $a=$ansi.GetString([System.IO.File]::ReadAllBytes($_.FullName))
       if ($u -ne $a) { "DIFFERS: " + $_.Name }
   }
   ```

2. **ไฟล์ `.json` และ `.md` ต้องมี BOM** เพื่อให้ Notepad รุ่นเก่าบน server อ่านภาษาไทยได้
   ถ้าแก้ด้วยเครื่องมือที่ไม่ใส่ BOM ให้สั่งซ้ำ

   ```powershell
   $enc = New-Object System.Text.UTF8Encoding($true)
   $t=[System.IO.File]::ReadAllText($f,[System.Text.Encoding]::UTF8)
   [System.IO.File]::WriteAllText($f,$t,$enc)
   ```

3. **ห้าม hardcode เกณฑ์เตือนในสคริปต์** ทุกค่าต้องอยู่ใน `settings.json` และอ่านผ่าน
   `Test-PMThreshold` เท่านั้น (พิสูจน์แล้วว่าใช้ได้จริง: ตั้ง `DiskFreePercent.Warn = 99`
   แล้วทุกไดรฟ์กลายเป็น WARN)

4. **ห้ามเขียนค่าอะไรลงเครื่องแม่ข่าย** เครื่องมือนี้ประกาศไว้ว่าอ่านอย่างเดียว
   ถ้าเพิ่มหัวข้อใหม่ต้องคงคุณสมบัตินี้ และระวังคำสั่งที่ "อ่าน" แต่ไปแตะไดรเวอร์หรือบริการระบบ
   (บทเรียนจาก `Get-NetAdapter`)

## เรื่องที่ตัดสินใจไปแล้ว ไม่ต้องรื้อ

| เรื่อง | ตัดสินใจว่า | เพราะ |
|---|---|---|
| `WU` (ตรวจ update ค้าง) | **ปิดเป็นค่าเริ่มต้น** | ทดสอบแล้วค้างเกิน 10 นาที เพราะต้องคุยกับ Microsoft Update |
| `CONN` (ping) | **ปิดเป็นค่าเริ่มต้น** | เป็นหัวข้อเดียวที่ส่งแพ็กเก็ตออกจริง เสี่ยงโดน IPS/IDS มองเป็นการสแกน |
| กราฟ `TREND` | ตัดสินสถานะด้วย **เปอร์เซ็นไทล์ที่ 95** ไม่ใช่ค่าสูงสุด | CPU กระชาก 100% วินาทีเดียวเป็นเรื่องปกติ ถ้าใช้ค่าสูงสุดจะเตือนผิดแทบทุกครั้ง |
| วันที่ในตาราง | ใช้ **ค.ศ.** รูปแบบ `yyyy-MM-dd` (หัวรายงานใช้ พ.ศ.) | ช่องตารางมีค่าเดียวแต่ผู้อ่านมีสองภาษา ใส่ 2569 ลอย ๆ จะกำกวม |
| ท้ายรายงาน | **ไม่มีช่องลงนาม** | ตามที่เลือกไว้ จะแปะเข้าหนังสือของหน่วยงานเอง |
