# หัวข้อที่ตรวจ, การตั้งค่า และการเพิ่มหัวข้อใหม่

ภาพรวมย่อและลิงก์ไปเอกสารอื่นอยู่ใน [README.md](README.md)

## หัวข้อที่ตรวจ

| รหัส | หัวข้อ | เกณฑ์เตือน (ปรับได้ใน `settings.json`) |
|---|---|---|
| `SYSTEM` | ข้อมูลระบบและเวลาทำงานต่อเนื่อง | เปิดต่อเนื่องเกิน 90 วัน |
| `DISK` | พื้นที่จัดเก็บข้อมูล | ว่าง < 20% หรือ < 20 GB (วิกฤตที่ 10%/10 GB) |
| `PERF` | หน่วยประมวลผลและหน่วยความจำ (ค่า ณ ขณะตรวจ) | CPU > 80% หรือหน่วยความจำว่าง < 15% |
| `TREND` | กราฟแนวโน้มตามช่วงเวลา | เปอร์เซ็นไทล์ที่ 95 เกินเกณฑ์เดียวกับ `PERF` (ต้องเก็บตัวอย่างก่อน — ดู [USAGE.md](USAGE.md)) |
| `PROC` | โปรเซสที่ใช้ทรัพยากรสูงสุด | ข้อมูลประกอบ ไม่มีเกณฑ์ |
| `SVC` | สถานะบริการ | บริการอัตโนมัติที่หยุดอยู่ / บริการใน watchlist ที่หยุด |
| `EVT` | บันทึกเหตุการณ์ | มี Error หรือ Critical ในช่วง 7 วัน |
| `HOTFIX` | ประวัติการติดตั้งรายการปรับปรุง | ไม่ได้ patch เกิน 60 วัน |
| `AV` | ระบบป้องกันไวรัส | ปิดการป้องกันเรียลไทม์ / ฐานข้อมูลเก่าเกิน 3 วัน |
| `BACKUP` | การสำรองข้อมูล | สำรองล่าสุดเกิน 7 วัน หรือมีรายการล้มเหลว |
| `CERT` | ใบรับรองอิเล็กทรอนิกส์ | หมดอายุใน 30 วัน (วิกฤตที่ 7 วัน) |
| `TIME` | การเทียบเวลามาตรฐาน | เครื่องไม่ได้ sync เวลา |
| `NET` | อุปกรณ์เครือข่ายและการตั้งค่า | ไม่มีการ์ดใดกำหนด DNS หรือ Gateway เลย |
| `CONN` | การเชื่อมต่อเครือข่าย | ping ไม่ผ่าน — **ปิดไว้เป็นค่าเริ่มต้น** |
| `REBOOT` | สถานะการรอเริ่มระบบใหม่ | มีรายการค้างรอรีสตาร์ท |
| `WU` | รายการปรับปรุงที่ค้างติดตั้ง | **ปิดไว้เป็นค่าเริ่มต้น** |
| `AGS` | สถานะไซต์ ArcGIS Server (เวอร์ชัน + เครื่องในไซต์) | เครื่องในไซต์ไม่ทำงาน — **ปิดไว้เป็นค่าเริ่มต้น ต้องตั้งค่าการเชื่อมต่อก่อน** |
| `AGSSVC` | บริการแผนที่: สถานะ อินสแตนซ์ จำนวนคำขอ เวลาประมวลผลเฉลี่ย | บริการที่ควรทำงานแต่หยุด / อินสแตนซ์เต็ม — **ปิดไว้เป็นค่าเริ่มต้น** |
| `AGSUSAGE` | รายงานการใช้งานที่มีอยู่แล้วบนไซต์: คำขอสะสม เวลาตอบสนองสูงสุด คำขอหมดเวลา | คำขอหมดเวลา > 0 — **ปิดไว้เป็นค่าเริ่มต้น ยังไม่เคยตรวจสอบ schema กับไซต์จริง** |

## ArcGIS Server

หัวข้อรหัส `AGS` ติดต่อ **ArcGIS Server Administrator API** จึงต้องตั้งค่าการเชื่อมต่อก่อนใช้งาน
และถูกปิดไว้เป็นค่าเริ่มต้นด้วยเหตุผลเดียวกับ `WU` และ `CONN` คือเป็นหัวข้อที่ติดต่อออกนอกเครื่อง

### ตั้งค่าการเชื่อมต่อ

ดับเบิลคลิก `Run-PM.cmd` (หรือไฟล์เดียว) แล้วเลือก **`[A]  ArcGIS Server connection...`**
จากนั้นเลือก `[1]` เพื่อกรอกข้อมูล เมนูจะแสดงตัวอย่างให้ครบทั้ง URL และชื่อบัญชี

**ที่อยู่ไซต์** — กรอกโดย**ไม่ต้องมี `/admin`** ต่อท้าย (ถ้าติดมาระบบจะตัดให้เอง)

| รูปแบบที่กรอก | ระบบจะแปลงเป็น |
|---|---|
| `https://gis.example.go.th/server` | เหมือนเดิม (ผ่าน Web Adaptor — พบบ่อยที่สุด) |
| `https://gisserver.example.go.th:6443` | `.../arcgis` (ต่อตรงเข้าเซิร์ฟเวอร์) |
| `https://localhost:6443` | `.../arcgis` (รันบนตัวเซิร์ฟเวอร์เอง) |

**บัญชีผู้ใช้** — ควรใช้บัญชีที่**อ่านอย่างเดียว** ไม่ต้องใช้ primary site administrator
เพราะ PMtools อ่านค่าเท่านั้น เช่น `pmreader`, `siteadmin`, หรือ `DOMAIN\gis_monitor`
ถ้าไซต์ใช้บัญชีของ Windows

### รหัสผ่านเก็บไว้ที่ไหน

เก็บที่ `%LOCALAPPDATA%\PMtools\arcgis-connection.xml` **ไม่ได้เก็บใน `Config\settings.json`**
เพราะไฟล์นั้นขึ้น git และถูกดึงออกมาด้วย `-ExtractConfig`

รหัสผ่านเข้ารหัสด้วย **Windows DPAPI** ซึ่งผูกกับบัญชีผู้ใช้และเครื่องที่ตั้งค่า —
คัดลอกไฟล์ไปเครื่องอื่นหรือเปิดด้วยบัญชีอื่นจะถอดรหัสไม่ได้ ถ้าเจอข้อความว่าถอดรหัสไม่ได้
ให้ตั้งค่าใหม่จากบัญชีที่จะใช้รันจริง

> เก็บไว้ใน profile ไม่ใช่ใน `Config\` เพราะแบบไฟล์เดียวแตกตัวลงโฟลเดอร์ชั่วคราวที่ถูกลบทุกครั้ง
> ถ้าเก็บข้าง `settings.json` จะหายทุกรอบ

### สั่งรัน

```powershell
.\Start-PMCheck.ps1 -Only AGS,AGSSVC,AGSUSAGE     # รันเฉพาะหัวข้อ ArcGIS
```

หรือลบ `"AGS"`, `"AGSSVC"` และ `"AGSUSAGE"` ออกจาก `Checks.Disabled` ใน `Config\settings.json` เพื่อให้รันทุกครั้ง

### `AGSSVC` — บริการแผนที่

อ่านจาก `/admin/services/{folder}/report` ซึ่งเป็น **GET ล้วน** ให้ทั้งสถานะบริการ จำนวนอินสแตนซ์
จำนวนคำขอสะสม และเวลาที่ใช้ประมวลผล

**เกณฑ์การตัดสินดูที่ configured เทียบกับ realTime ไม่ใช่ดูแค่ว่า "ทำงานอยู่ไหม"**

| configured | realTime | ผล | เหตุผล |
|---|---|---|---|
| STARTED | STARTED | ปกติ | |
| STARTED | STOPPED | **วิกฤต** | ตั้งให้ทำงานแต่ไม่ทำงาน = พังจริง |
| STOPPED | STOPPED | ข้อมูล | **ตั้งใจปิด ไม่ใช่ปัญหา** |
| STOPPED | STARTED | ควรเฝ้าระวัง | ทำงานทั้งที่ตั้งให้หยุด |
| อินสแตนซ์ busy = max | | ควรเฝ้าระวัง | คำขอใหม่ต้องรอคิว |

> ArcGIS ติดตั้งมาพร้อมบริการในโฟลเดอร์ `System` และ `Utilities` ที่**ปิดไว้เป็นค่าเริ่มต้นหลายสิบรายการ**
> ไซต์จริงที่ทดสอบมี 16 รายการ ถ้าตัดสินว่า "ไม่ทำงาน = วิกฤต" จะขึ้นเตือนผิด 16 รายการทันที

### จำนวนคำขอและเวลาตอบสนอง

| ต้องการ | ได้จาก | หมายเหตุ |
|---|---|---|
| จำนวนคำขอสะสมรายบริการ | `transactions` | สะสมตั้งแต่บริการเริ่มทำงานล่าสุด |
| เวลาประมวลผลเฉลี่ย | `totalBusyTime / transactions` | **เวลาที่ฝั่งเซิร์ฟเวอร์ใช้ประมวลผล ไม่รวมเวลารอคิวและเวลาเดินทางบนเครือข่าย** |
| อินสแตนซ์ที่ใช้อยู่ | `busy / max` | `max = 0` แปลว่าใช้ shared instance pool |

**ค่าเฉลี่ยเวลาตอบสนองแบบเต็ม (RequestAvgResponseTime) ไม่มีในหัวข้อนี้** เพราะอยู่ใน usage report
ซึ่งเป็นข้อมูลของหัวข้อ `AGSUSAGE` ด้านล่างแทน

### `AGSUSAGE` — รายงานการใช้งาน (คำขอสะสม เวลาตอบสนอง คำขอหมดเวลา)

อ่านจาก `/admin/usagereports/<ชื่อรายงาน>/data` ซึ่งเป็น **GET ล้วน** แต่ทำได้เฉพาะรายงานที่
**มีอยู่แล้ว**บนไซต์ — การสร้างรายงานใหม่คือการเขียนและหัวข้อนี้**ไม่ทำเด็ดขาด** ArcGIS Server
Manager มักสร้างรายงานชื่อ `System` ให้อัตโนมัติตั้งแต่ครั้งแรกที่มีคนเปิดหน้าสถิติ ไซต์ที่ไม่เคย
เปิดหน้านั้นเลยจะไม่มีรายงาน และหัวข้อนี้จะลดระดับเป็นการ์ดข้อมูล (`INFO`) ไม่ใช่ error

| ค่าที่แสดง | ได้จากตัวชี้วัด | หมายเหตุ |
|---|---|---|
| จำนวนคำขอสะสม | `RequestCount` (รวมทุกช่วงเวลาในรายงาน) | ครอบคลุมทั้งไซต์ ต่างจาก `AGSSVC` ที่แยกรายบริการ |
| เวลาตอบสนองสูงสุด | `RequestMaxResponseTime` (ค่าสูงสุดในรายงาน) | เวลาตอบสนองปลายทางจริง รวมเวลารอคิวและเครือข่าย ต่างจากเวลาประมวลผลของ `AGSSVC` |
| คำขอที่หมดเวลา | `RequestsTimedOut` (รวมทุกช่วงเวลาในรายงาน) | มากกว่า 0 รายการขึ้นเป็นวิกฤต/เฝ้าระวังตาม `Thresholds.AGSUsageTimedOutRequests` |

ถ้าไซต์มีมากกว่าหนึ่งรายงาน จะอ่านรายงานแรกที่ API คืนมา (ปกติมีรายงานเดียว) ชื่อรายงานที่ใช้จริง
แสดงอยู่ในผลลัพธ์เสมอเพื่อไม่ให้เป็นการเลือกแบบเงียบ ๆ

> **ยังไม่เคยตรวจสอบรูปแบบข้อมูลนี้กับไซต์จริง** — ตัวเลขที่เคยอ่านได้จากไซต์ทดสอบ (คำขอ 112,811
> ครั้ง เวลาตอบสนองสูงสุด 561.1 มิลลิวินาที) ยืนยันแค่ว่า endpoint นี้อ่านได้ ไม่ได้ยืนยันโครงสร้าง
> JSON แบบเต็มของ `report.metadata.metrics` และ `report.'report-data'` ที่โค้ดคาดไว้ จึงปิดไว้เป็น
> ค่าเริ่มต้นเหมือน `AGS`/`AGSSVC` ตอนเริ่มพัฒนา ทดสอบด้วย mock ครบ 6 สถานการณ์แล้ว (ปกติ, มีคำขอ
> หมดเวลา, ไม่มีรายงาน, อ่านข้อมูลไม่ได้, มีรายงานแต่ไม่มีค่าที่อ่านได้, รหัสผ่านผิด) แต่ยังไม่เคยรัน
> กับไซต์จริง — ดู `HANDOVER.md` ก่อนเปิดใช้งานจริง

### ค่าที่ปรับได้ (`Config\settings.json` หัวข้อ `ArcGIS`)

| ค่า | ทำอะไร |
|---|---|
| `IgnoreCertificateErrors` | ค่าเริ่มต้น `true` เพราะ ArcGIS Server ใช้ใบรับรองที่เซ็นเองบนพอร์ต 6443 ถ้าไซต์ใช้ใบรับรองที่เชื่อถือได้แล้วตั้งเป็น `false` |
| `TokenExpirationMinutes` | อายุ token ค่าเริ่มต้น 60 |
| `TimeoutSeconds` | เวลารอตอนขอ token ค่าเริ่มต้น 30 |
| `RequestTimeoutSeconds` | เวลารอตอนเรียก API ค่าเริ่มต้น 60 |

### ยังอ่านอย่างเดียวเหมือนเดิม

ทุกหัวข้อ ArcGIS ใช้ `GET` ยกเว้น `generateToken` ที่เป็น `POST` **เพราะ Admin API บังคับให้
token เดินทางใน request body ไม่ใช่เพราะมันเขียนอะไร**

สิ่งที่**จงใจไม่ทำ**คือการ*สร้าง*รายงานการใช้งาน — `AGSUSAGE` อ่านได้เฉพาะรายงานที่มีอยู่แล้ว
บนไซต์ผ่าน `/usagereports` และ `/usagereports/<ชื่อ>/data` ซึ่งเป็น `GET` ล้วน แต่จะไม่สั่งสร้าง
รายงานใหม่เด็ดขาด เพราะการสร้างคือการเขียน ไซต์ที่ไม่เคยมีใครเปิดหน้าสถิติของ Manager จะไม่มี
รายงานให้อ่านเลย และหัวข้อนี้จะรายงานว่า "ไม่มีข้อมูล" แทนที่จะพยายามสร้างให้เอง

## การตั้งค่า

### `Config\settings.json`

ชื่อหน่วยงาน ผู้ตรวจสอบ และ **เกณฑ์เตือนทุกค่า** อยู่ในไฟล์นี้ที่เดียว
ไม่มีเกณฑ์ไหนถูกเขียนตายไว้ในสคริปต์ แก้ที่นี่แล้วมีผลทันที

```json
"DiskFreePercent": { "Warn": 20, "Crit": 10, "Direction": "Lower" }
```

`Direction` บอกว่าค่ามาก/น้อยแบบไหนคือแย่ — `Lower` คือยิ่งน้อยยิ่งแย่ (พื้นที่ว่าง, วันคงเหลือก่อนหมดอายุ)
ส่วน `Higher` คือยิ่งมากยิ่งแย่ (การใช้งาน CPU, จำนวนวันที่ไม่ได้ patch)

**หัวข้อข้อเสนอแนะ** — ปิดไว้เป็นค่าเริ่มต้นเพื่อให้รายงานกระชับ เปิดกลับได้ที่

```json
"Report": { "ShowRecommendations": true }
```

เมื่อปิด รายงานจะเหลือ 3 ส่วน คือ สรุปผลสำหรับผู้บริหาร / ภาพรวมรายหัวข้อ / รายละเอียดผลการตรวจสอบ
โดยช่อง "สรุปย่อ" ในตารางภาพรวมยังบอกอยู่ว่าแต่ละหัวข้อพบอะไร
และ **ข้อเสนอแนะฉบับเต็มยังถูกเก็บไว้ใน `PM-Data.json` ครบเสมอ** ไม่ว่าจะเปิดหรือปิด
จึงย้อนไปดูได้ตลอดโดยไม่ต้องรันใหม่

### `Config\services.json`

ใส่ชื่อบริการสำคัญของระบบนั้น ๆ ลงใน `Watchlist` ถ้าบริการในรายการหยุดทำงานจะถูกจัดเป็น **วิกฤต**

ถ้าปล่อยว่างไว้ก็ใช้งานได้ทันที เพราะเครื่องมือจะตรวจหาบริการที่ตั้งค่าเริ่มอัตโนมัติ
แต่หยุดอยู่ให้เองอยู่แล้ว (จัดเป็นระดับควรเฝ้าระวัง) ในไฟล์มีตัวอย่างชื่อบริการที่พบบ่อยแยกตาม role ให้คัดลอกไปใช้

> ใส่ **ชื่อบริการ** ไม่ใช่ชื่อที่แสดง ดูชื่อจริงได้ด้วย
> `Get-Service | Select-Object Name, DisplayName, Status`

### `Config\i18n.json`

ข้อความทุกคำที่ปรากฏในรายงานอยู่ในไฟล์นี้ ทั้งภาษาไทยและอังกฤษ
ถ้าต้องการเปลี่ยนสำนวนให้ตรงกับที่หน่วยงานใช้ แก้ที่นี่ได้เลยโดยไม่ต้องแตะสคริปต์
เครื่องหมาย `{0}` `{1}` คือช่องเติมค่า **ห้ามลบหรือสลับลำดับ**

## การเพิ่มหัวข้อตรวจใหม่

โครงสร้างเป็นแบบ **1 ไฟล์ = 1 หัวข้อ** เพิ่มหัวข้อใหม่ได้โดยไม่ต้องแก้ไฟล์เดิมเลย

1. สร้างไฟล์ใหม่ใน `Checks\` ตั้งชื่อขึ้นต้นด้วยตัวเลขเพื่อกำหนดลำดับ เช่น `98-MyCheck.ps1`
2. เขียนฟังก์ชันที่คืนค่าด้วย `New-PMResult` (ดูรูปแบบจาก `20-DiskSpace.ps1` เป็นตัวอย่างที่สั้นที่สุด)
3. ปิดท้ายไฟล์ด้วย `Register-PMCheck -Id ... -TitleKey ... -Function ...`
4. เพิ่มข้อความไทย/อังกฤษที่ใช้ลงใน `Config\i18n.json`

ตัวสร้างรายงานไม่จำเป็นต้องรู้จักหัวข้อใหม่เลย เพราะทุกหัวข้อคืนค่าโครงสร้างเดียวกัน
เหตุผลเชิงออกแบบเบื้องหลังโครงสร้างนี้ดูเพิ่มได้ใน [HANDOVER.md](HANDOVER.md)
