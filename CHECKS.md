## หัวข้อที่ตรวจ

| รหัส | หัวข้อ | เกณฑ์เตือน (ปรับได้ใน `settings.json`) |
|---|---|---|
| `SYSTEM` | ข้อมูลระบบและเวลาทำงานต่อเนื่อง | เปิดต่อเนื่องเกิน 90 วัน |
| `DISK` | พื้นที่จัดเก็บข้อมูล | ว่าง < 20% หรือ < 20 GB (วิกฤตที่ 10%/10 GB) |
| `PERF` | หน่วยประมวลผลและหน่วยความจำ (ค่า ณ ขณะตรวจ) | CPU > 80% หรือหน่วยความจำว่าง < 15% |
| `TREND` | กราฟแนวโน้มตามช่วงเวลา | เปอร์เซ็นไทล์ที่ 95 เกินเกณฑ์เดียวกับ `PERF` (ต้องเก็บตัวอย่างก่อน — ดู [USAGE.md](USAGE.md)) |
| `PROC` | โปรเซสที่ใช้ทรัพยากรสูงสุด | ข้อมูลประกอบ ไม่มีเกณฑ์ |
| `SVC` | สถานะบริการ | บริการอัตโนมัติที่หยุดอยู่ / บริการใน watchlist ที่หยุด |
| `EVT` | บันทึกเหตุการณ์ | มี Error หรือ Critical ในช่วง 7 วัน |
| `HOTFIX` | ประวัติการติดตั้งรายการปรับปรุง | ไม่ได้ patch เกิน 60 วัน |
| `AV` | ระบบป้องกันไวรัส | ปิดการป้องกันเรียลไทม์ / ฐานข้อมูลเก่าเกิน 3 วัน |
| `BACKUP` | การสำรองข้อมูล | สำรองล่าสุดเกิน 7 วัน หรือมีรายการล้มเหลว |
| `CERT` | ใบรับรองอิเล็กทรอนิกส์ | หมดอายุใน 30 วัน (วิกฤตที่ 7 วัน) |
| `TIME` | การเทียบเวลามาตรฐาน | เครื่องไม่ได้ sync เวลา |
| `NET` | อุปกรณ์เครือข่ายและการตั้งค่า | ไม่มีการ์ดใดกำหนด DNS หรือ Gateway เลย |
| `CONN` | การเชื่อมต่อเครือข่าย | ping ไม่ผ่าน — **ปิดไว้เป็นค่าเริ่มต้น** |
| `REBOOT` | สถานะการรอเริ่มระบบใหม่ | มีรายการค้างรอรีสตาร์ท |
| `WU` | รายการปรับปรุงที่ค้างติดตั้ง | **ปิดไว้เป็นค่าเริ่มต้น** |
| `AGS` | สถานะไซต์ ArcGIS Server (เวอร์ชัน + เครื่องในไซต์) | เครื่องในไซต์ไม่ทำงาน — **ปิดไว้เป็นค่าเริ่มต้น ต้องตั้งค่าการเชื่อมต่อก่อน** |
| `AGSSVC` | บริการแผนที่: สถานะ อินสแตนซ์ จำนวนคำขอ เวลาประมวลผลเฉลี่ย | บริการที่ควรทำงานแต่หยุด / อินสแตนซ์เต็ม — **ปิดไว้เป็นค่าเริ่มต้น** |
| `AGSUSAGE` | รายงานการใช้งานที่มีอยู่แล้วบนไซต์: คำขอสะสม เวลาตอบสนองสูงสุด คำขอหมดเวลา | คำขอหมดเวลา > 0 — **ปิดไว้เป็นค่าเริ่มต้น ต้องตั้งค่าการเชื่อมต่อก่อน** |
| `AGSDATA` | การเชื่อมต่อฐานข้อมูลที่ลงทะเบียนไว้กับไซต์ (enterprise geodatabase, file share ฯลฯ) | เชื่อมต่อไม่ได้แม้แต่รายการเดียว — **ปิดไว้เป็นค่าเริ่มต้น ต้องตั้งค่าการเชื่อมต่อก่อน** |
| `AGSLOG` | บันทึกเหตุการณ์ของ ArcGIS Server ระดับ SEVERE/WARNING ย้อน 7 วัน (สูงสุด 100 รายการต่อระดับ) | พบบันทึกระดับ SEVERE แม้แต่รายการเดียว — **ปิดไว้เป็นค่าเริ่มต้น ต้องตั้งค่าการเชื่อมต่อก่อน** |

## ArcGIS Server

หัวข้อรหัส `AGS` ติดต่อ **ArcGIS Server Administrator API** จึงต้องตั้งค่าการเชื่อมต่อก่อนใช้งาน
และถูกปิดไว้เป็นค่าเริ่มต้นด้วยเหตุผลเดียวกับ `WU` และ `CONN` คือเป็นหัวข้อที่ติดต่อออกนอกเครื่อง

### ตั้งค่าการเชื่อมต่อ

ดับเบิลคลิก `Run-PM.cmd` (หรือไฟล์เดียว) แล้วเลือก **`[A]  ArcGIS Server connection...`**
จากนั้นเลือก `[1]` เพื่อกรอกข้อมูล เมนูจะแสดงตัวอย่างให้ครบทั้ง URL และชื่อบัญชี

**ที่อยู่ไซต์** — กรอกโดย**ไม่ต้องมี `/admin`** ต่อท้าย (ถ้าติดมาระบบจะตัดให้เอง)

| รูปแบบที่กรอก | ระบบจะแปลงเป็น |
|---|---|
| `https://gis.example.go.th/server` | เหมือนเดิม (ผ่าน Web Adaptor — พบบ่อยที่สุด) |
| `https://gisserver.example.go.th:6443` | `.../arcgis` (ต่อตรงเข้าเซิร์ฟเวอร์) |
| `https://localhost:6443` | `.../arcgis` (รันบนตัวเซิร์ฟเวอร์เอง) |

**บัญชีผู้ใช้** — ควรใช้บัญชีที่**อ่านอย่างเดียว** ไม่ต้องใช้ primary site administrator
เพราะ PMtools อ่านค่าเท่านั้น เช่น `pmreader`, `siteadmin`, หรือ `DOMAIN\gis_monitor`
ถ้าไซต์ใช้บัญชีของ Windows

### รหัสผ่านเก็บไว้ที่ไหน

เก็บที่ `%LOCALAPPDATA%\PMtools\arcgis-connection.xml` **ไม่ได้เก็บใน `Config\settings.json`**
เพราะไฟล์นั้นขึ้น git และถูกดึงออกมาด้วย `-ExtractConfig`

รหัสผ่านเข้ารหัสด้วย **Windows DPAPI** ซึ่งผูกกับบัญชีผู้ใช้และเครื่องที่ตั้งค่า —
คัดลอกไฟล์ไปเครื่องอื่นหรือเปิดด้วยบัญชีอื่นจะถอดรหัสไม่ได้ ถ้าเจอข้อความว่าถอดรหัสไม่ได้
ให้ตั้งค่าใหม่จากบัญชีที่จะใช้รันจริง

> เก็บไว้ใน profile ไม่ใช่ใน `Config\` เพราะแบบไฟล์เดียวแตกตัวลงโฟลเดอร์ชั่วคราวที่ถูกลบทุกครั้ง
> ถ้าเก็บข้าง `settings.json` จะหายทุกรอบ

### สั่งรัน

```powershell
.\Start-PMCheck.ps1 -Group ArcGIS                                # ครบทั้งห้าหัวข้อ ArcGIS ในคำสั่งเดียว บายพาส Checks.Disabled
.\Start-PMCheck.ps1 -Only AGS,AGSSVC,AGSUSAGE,AGSDATA,AGSLOG    # เทียบเท่ากัน เจาะจงทีละ id
```

**`-Group Server`/`-Group ArcGIS` แยกสองกลุ่มออกจากกันเด็ดขาด** — เมนูหลัก (`Run-PM.cmd`) ก็ถาม
ตั้งแต่หน้าแรกว่าจะตรวจ Server หรือ ArcGIS Server เพราะเครื่องแม่ข่ายทั่วไปกับไซต์ ArcGIS Server
ไม่มีเหตุผลต้องตรวจรวมกันในรอบเดียว `-Only` ที่ผสมหัวข้อ Server กับ ArcGIS
(เช่น `-Only DISK,AGS`) จะถูกปฏิเสธพร้อมข้อความบอกให้รันแยกสองรอบแทน ไม่ว่าจะระบุ `-Group` หรือไม่

หรือลบ `"AGS"`, `"AGSSVC"`, `"AGSUSAGE"`, `"AGSDATA"` และ `"AGSLOG"` ออกจาก `Checks.Disabled` ใน
`Config\settings.json` เพื่อให้รันทุกครั้ง (ใช้กับการรันแบบไม่ระบุ `-Group`/`-Only` เลย)

### `AGSSVC` — บริการแผนที่

อ่านจาก `/admin/services/{folder}/report` ซึ่งเป็น **GET ล้วน** ให้ทั้งสถานะบริการ จำนวนอินสแตนซ์
จำนวนคำขอสะสม และเวลาที่ใช้ประมวลผล

**เกณฑ์การตัดสินดูที่ configured เทียบกับ realTime ไม่ใช่ดูแค่ว่า "ทำงานอยู่ไหม"**

| configured | realTime | ผล | เหตุผล |
|---|---|---|---|
| STARTED | STARTED | ปกติ | |
| STARTED | STOPPED | **วิกฤต** | ตั้งให้ทำงานแต่ไม่ทำงาน = พังจริง |
| STOPPED | STOPPED | ข้อมูล | **ตั้งใจปิด ไม่ใช่ปัญหา** |
| STOPPED | STARTED | ควรเฝ้าระวัง | ทำงานทั้งที่ตั้งให้หยุด |
| อินสแตนซ์ busy = max | | ควรเฝ้าระวัง | คำขอใหม่ต้องรอคิว |

> ArcGIS ติดตั้งมาพร้อมบริการในโฟลเดอร์ `System` และ `Utilities` ที่**ปิดไว้เป็นค่าเริ่มต้นหลายสิบรายการ**
> ไซต์จริงที่ทดสอบมี 16 รายการ ถ้าตัดสินว่า "ไม่ทำงาน = วิกฤต" จะขึ้นเตือนผิด 16 รายการทันที

### จำนวนคำขอและเวลาตอบสนอง

| ต้องการ | ได้จาก | หมายเหตุ |
|---|---|---|
| จำนวนคำขอสะสมรายบริการ | `transactions` | สะสมตั้งแต่บริการเริ่มทำงานล่าสุด |
| เวลาประมวลผลเฉลี่ย | `totalBusyTime / transactions` | **เวลาที่ฝั่งเซิร์ฟเวอร์ใช้ประมวลผล ไม่รวมเวลารอคิวและเวลาเดินทางบนเครือข่าย** |
| อินสแตนซ์ที่ใช้อยู่ | `busy / max` | `max = 0` แปลว่าใช้ shared instance pool |

**ค่าเฉลี่ยเวลาตอบสนองแบบเต็ม (RequestAvgResponseTime) ไม่มีในหัวข้อนี้** เพราะอยู่ใน usage report
ซึ่งเป็นข้อมูลของหัวข้อ `AGSUSAGE` ด้านล่างแทน

### `AGSUSAGE` — รายงานการใช้งาน (คำขอสะสม เวลาตอบสนอง คำขอหมดเวลา)

อ่านจาก `/admin/usagereports/<ชื่อรายงาน>/data` ซึ่งเป็น **GET ล้วน** แต่ทำได้เฉพาะรายงานที่
**มีอยู่แล้ว**บนไซต์ — การสร้างรายงานใหม่คือการเขียนและหัวข้อนี้**ไม่ทำเด็ดขาด**

**Schema ยืนยันกับไซต์จริงแล้ว (23 ก.ค. 2569, ArcGIS Server 11.5.0)** ต่างจากที่คาดไว้ตอนแรกสามจุด:

1. `GET /admin/usagereports` คืนรายชื่อรายงานภายใต้คีย์ `metrics` ไม่ใช่ `usagereports`
2. ArcGIS Server Manager สร้างรายงาน**ชั่วคราว**ทุกครั้งที่มีคนเปิดหน้า Manager -> Server ->
   Statistics (`metadata.temp = true`, ชื่อเป็นตัวเลข timestamp, หมดอายุเอง) แยกจากรายงาน
   **ถาวร** ที่สร้างไว้ครั้งแรกและอยู่ต่อ (`metadata.temp = false`, ชื่อเป็นข้อความคงที่ เช่น
   `Total requests for the last 7 days`) หัวข้อนี้อ่านเฉพาะรายงานถาวรเท่านั้น และแต่ละรายงาน
   ครอบคลุม**ตัวชี้วัดเดียว** ไม่ใช่สามตัวชี้วัดในรายงานเดียวอย่างที่เคยคาด
3. เรียก `/data` เฉย ๆ ไม่พอ **ต้องส่งพารามิเตอร์ `filter`** (นำ `since` และ `queries` ของรายงาน
   นั้นจาก `/usagereports` มาห่อเป็น JSON ส่งกลับไป) ไม่งั้นเซิร์ฟเวอร์ตอบ HTTP 500 ภายใน
   (`RequestUtil.getParameterIgnoreCase(...) is null`) ซึ่งไม่ได้แปลว่ารายงานไม่มีอยู่

| ค่าที่แสดง | ได้จากตัวชี้วัด | หมายเหตุ |
|---|---|---|
| จำนวนคำขอสะสม | `RequestCount` (รวมทุกช่วงเวลาในรายงาน) | ครอบคลุมทั้งไซต์ ต่างจาก `AGSSVC` ที่แยกรายบริการ |
| เวลาตอบสนองสูงสุด | `RequestMaxResponseTime` (ค่าสูงสุดในรายงาน) | เวลาตอบสนองปลายทางจริง รวมเวลารอคิวและเครือข่าย ต่างจากเวลาประมวลผลของ `AGSSVC` |
| คำขอที่หมดเวลา | `RequestsTimedOut` (รวมทุกช่วงเวลาในรายงาน) | มากกว่า 0 รายการขึ้นเป็นวิกฤต/เฝ้าระวังตาม `Thresholds.AGSUsageTimedOutRequests` |

อ่านทุกรายงานถาวรที่เจอ (ไซต์ทดสอบมี 3 รายงาน หนึ่งรายงานต่อหนึ่งตัวชี้วัด) แล้วรวมผลตามชื่อ
ตัวชี้วัดที่แต่ละแถวข้อมูลติดป้ายตัวเองไว้ (`metric-type`) ชื่อรายงานที่อ่านได้ทั้งหมดแสดงอยู่ใน
ผลลัพธ์เสมอ ถ้าไซต์ไม่มีรายงานถาวรเลยจะลดระดับเป็นการ์ดข้อมูล (`INFO`) ไม่ใช่ error

ทดสอบกับไซต์จริงแล้ว ตัวเลขตรงกับที่เคยยืนยันด้วยมือไว้ก่อนหน้า (คำขอ 7 วันล่าสุด ~112,000 ครั้ง
เวลาตอบสนองสูงสุด 561.1 มิลลิวินาที คำขอหมดเวลา 0 — ตัวเลขคำขอต่างกันเล็กน้อยระหว่างสองรอบเพราะ
เป็นหน้าต่างเวลาแบบเลื่อน 7 วันล่าสุดตลอด ไม่ใช่ช่วงคงที่)

### กราฟจำนวนคำขอตามช่วงเวลา (23 ก.ค. 2569)

หัวข้อนี้ยังเก็บ `RequestCount` แยกตามช่วงเวลาย่อยในรายงาน (`time-slices` — ไซต์ทดสอบแบ่งเป็นช่วงละ
4 ชั่วโมง ครอบคลุม 7 วันล่าสุด รวม 42 จุดข้อมูล) แล้ววาดเป็นกราฟเส้นในรายงาน HTML **ไม่ใช่แค่ผลรวม
เดียวเหมือนแถวข้อมูลด้านบน** — ใช้ข้อมูลที่ดึงมาอยู่แล้วในการเรียกเดิม ไม่มีการเรียก endpoint เพิ่ม
และเป็นภาพรวมทั้งไซต์เหมือนตัวเลขอื่นในหัวข้อนี้ **ไม่แยกรายบริการแผนที่** (การแยกรายบริการต้อง
สแนปช็อตตัวนับสะสมของ `AGSSVC` ซ้ำ ๆ ข้ามหลายรอบการรัน ซึ่งยังไม่ได้ทำในรุ่นนี้)

กราฟจะวาดก็ต่อเมื่อมีอย่างน้อย 2 จุดข้อมูล ถ้าน้อยกว่านั้นจะไม่แสดงกราฟแต่แถวข้อมูลอื่นยังครบตามปกติ

### `AGSDATA` — การเชื่อมต่อฐานข้อมูลที่ลงทะเบียนไว้

ตรวจว่าฐานข้อมูล/แหล่งข้อมูลที่ลงทะเบียนไว้กับไซต์ (enterprise geodatabase, file share, cloud
store, big data file share, NoSQL database, raster store) ยังเชื่อมต่อได้จริงหรือไม่ ทีละรายการ

**ไม่ใช้ `/data/validateAllDataItems`** ตามที่เอกสาร Esri แนะนำ เพราะสำรวจกับไซต์จริงแล้วพบว่า
endpoint นี้คืนแค่ `{"status":"success"}` รวมทั้งไซต์ ไม่บอกว่ารายการไหนเชื่อมต่อได้หรือไม่ได้ —
ใช้ `POST /data/findItems` ดึงรายการทั้งหมดก่อน (คำสั่งเดียว ไม่ต้องไล่ทีละหมวด) แล้ว
`POST /data/validateDataItem` ทดสอบทีละรายการแทน ซึ่งตอบสถานะเป็นรายการจริง

> **คำเตือนด้านความปลอดภัยที่สำคัญ** — `/data/findItems` คืนฟิลด์ `info.connectionString` ของ
> enterprise geodatabase มาด้วยเสมอ ซึ่งมี**รหัสผ่านที่เข้ารหัสไว้ (`ENCRYPTED_PASSWORD`)** รวมถึง
> ชื่อเซิร์ฟเวอร์ ชื่อฐานข้อมูล และชื่อบัญชี ก้อนข้อมูลนี้ใช้แค่ในหน่วยความจำเพื่อสร้างคำขอ
> `validateDataItem` ของรายการเดียวกัน **ไม่ถูกเก็บลง `PM-Data.json` หรือแสดงในรายงานเลย** —
> สิ่งที่บันทึกไว้มีแค่ path, ประเภท (`egdb`/`folder`/ฯลฯ) และผลเชื่อมต่อได้/ไม่ได้เท่านั้น

| ผล | ความหมาย |
|---|---|
| เชื่อมต่อได้ปกติ | ตอบ `{"status":"success"}` |
| เชื่อมต่อไม่ได้ (วิกฤต) | ตอบ `{"status":"error", ...}` — บริการแผนที่ที่ใช้แหล่งข้อมูลนี้อาจใช้งานไม่ได้หรือแสดงข้อมูลไม่ครบ |

ถ้าไซต์ไม่มีรายการลงทะเบียนไว้เลยจะลดระดับเป็นการ์ดข้อมูล (`INFO`) ถ้าอ่านรายการทั้งหมดไม่ได้
(เช่น สิทธิ์บัญชีไม่พอ) จะลดระดับเป็น `WARN` แทนที่จะทำให้ทั้ง run พัง

### `AGSLOG` — บันทึกเหตุการณ์ระดับ SEVERE/WARNING

อ่านบันทึกเหตุการณ์ของ ArcGIS Server (`POST /admin/logs/query`) ย้อนหลังตามจำนวนวันที่ตั้งไว้
(`ArcGIS.LogLookbackDays` ค่าเริ่มต้น 7) นับและจัดกลุ่มรายการระดับ SEVERE และ WARNING แยกกัน

**`filter` เป็นพารามิเตอร์บังคับ** เหมือนที่เจอกับ `AGSUSAGE` — เรียกโดยไม่ส่ง `filter` จะได้ HTTP 500
`Invalid filter value. / missing query parameters` ค่าที่ใช้ได้คือ
`{"codes":[],"processIds":[],"users":[],"server":"*"}` (ว่างเปล่า = ไม่กรองอะไรเพิ่ม)

**`level` เป็นเกณฑ์ขั้นต่ำ ไม่ใช่ตัวกรองแบบตรงตัว** — ขอ `level=WARNING` จะได้ทั้ง WARNING และ SEVERE
ปนกันมา เรียงจากล่าสุดไปเก่าสุด **ปัญหาที่เจอจริงกับไซต์ทดสอบ**: มี WARNING ตัวเดียวที่เกิดถี่มาก
(บริการอ้างอิงไฟล์ข้อมูลที่ถูกลบไปแล้ว) จนเต็มหน้าแรกของผลลัพธ์ ถ้า query รวมกันครั้งเดียวจะบัง
SEVERE ตัวจริงที่เกิดก่อนหน้าไม่ให้โผล่มาเลย จึง**แยก query สองครั้ง** (`level=SEVERE` กับ
`level=WARNING`) และกรองผลลัพธ์แต่ละชุดด้วย `type` ของตัวเองอีกชั้น ไม่ให้นับซ้ำ

**ดึงมาสูงสุด 100 รายการต่อระดับ** (`ArcGIS.LogPageSize`) เพื่อความเร็ว — ไม่ใช่ตัวเลขสรุปที่ครบเสมอไป
ถ้า `hasMore` เป็นจริงจะมีการ์ดข้อมูลบอกว่าตัวเลขที่แสดงเป็นค่าอย่างน้อยที่สุด ไม่ใช่จำนวนที่ครบทั้งหมด
จัดกลุ่มรายการด้วย (เครื่อง, รหัส) เหมือนหัวข้อ `EVT` ตัดจำนวนกลุ่มที่แสดงด้วย `ArcGIS.LogMaxGroups`

> **กับดักที่เจอ** — endpoint นี้รายงานข้อผิดพลาดของตัวเองเป็น `{"status":"error","messages":[...]}`
> ในเนื้อหา HTTP 200 เหมือน `AGSDATA`/`AGSUSAGE` **ไม่ใช่รูปแบบ `{"error":{...}}`** ที่
> `Invoke-PMArcGISAdmin` ดักจับเป็น exception ให้อัตโนมัติ ถ้าตรวจแค่ `try/catch` เฉย ๆ จะพลาด
> เคสนี้ไปเงียบ ๆ — ทดสอบแล้วพบว่าเดิมจะรายงาน "ปกติ ไม่พบปัญหา" ทั้งที่ query จริงล้มเหลว
> (สิทธิ์บัญชีไม่พอ) แก้โดยตรวจ `$resp.status` เองอีกชั้น และถ้าอ่าน log ไม่ได้เลยทั้งสองระดับ
> จะขึ้น `WARN` "อ่านบันทึกไม่ได้" แทนที่จะขึ้น `OK` เงียบ ๆ

### ค่าที่ปรับได้ (`Config\settings.json` หัวข้อ `ArcGIS`)

| ค่า | ทำอะไร |
|---|---|
| `IgnoreCertificateErrors` | ค่าเริ่มต้น `true` เพราะ ArcGIS Server ใช้ใบรับรองที่เซ็นเองบนพอร์ต 6443 ถ้าไซต์ใช้ใบรับรองที่เชื่อถือได้แล้วตั้งเป็น `false` |
| `TokenExpirationMinutes` | อายุ token ค่าเริ่มต้น 60 |
| `TimeoutSeconds` | เวลารอตอนขอ token ค่าเริ่มต้น 30 |
| `RequestTimeoutSeconds` | เวลารอตอนเรียก API ค่าเริ่มต้น 60 |
| `LogLookbackDays` | จำนวนวันย้อนหลังที่ `AGSLOG` อ่านบันทึกเหตุการณ์ ค่าเริ่มต้น 7 |
| `LogPageSize` | จำนวนบันทึกสูงสุดที่ `AGSLOG` ดึงมาต่อระดับความรุนแรง ค่าเริ่มต้น 100 |
| `LogMaxGroups` | จำนวนกลุ่มสูงสุดที่ `AGSLOG` แสดงในตาราง ค่าเริ่มต้น 25 |

### ยังอ่านอย่างเดียวเหมือนเดิม

ทุกหัวข้อ ArcGIS ใช้ `GET` ยกเว้นสี่จุดที่ต้องเป็น `POST` เพราะ Admin API บังคับ **ไม่ใช่เพราะ
เขียนอะไร**: `generateToken` (รหัสผ่านต้องเดินทางใน request body), `data/findItems`/
`data/validateDataItem` ของ `AGSDATA` และ `logs/query` ของ `AGSLOG` (ทั้งสามหลังต้องส่ง
path/type/filter เป็น JSON ยาวเกินจะใส่ใน query string)

สิ่งที่**จงใจไม่ทำ**คือการ*สร้าง*รายงานการใช้งาน — `AGSUSAGE` อ่านได้เฉพาะรายงานที่มีอยู่แล้ว
บนไซต์ผ่าน `/usagereports` และ `/usagereports/<ชื่อ>/data` ซึ่งเป็น `GET` ล้วน แต่จะไม่สั่งสร้าง
รายงานใหม่เด็ดขาด เพราะการสร้างคือการเขียน ไซต์ที่ไม่เคยมีใครเปิดหน้าสถิติของ Manager จะไม่มี
รายงานให้อ่านเลย และหัวข้อนี้จะรายงานว่า "ไม่มีข้อมูล" แทนที่จะพยายามสร้างให้เอง

`AGSDATA` ก็เช่นกัน — `data/validateDataItem` แค่**ทดสอบ**การเชื่อมต่อด้วย credential ที่ลงทะเบียน
ไว้แล้ว ไม่ได้แก้ไขหรือลงทะเบียนอะไรใหม่ และไม่เคยรับหรือส่งรหัสผ่านของฐานข้อมูลเอง (ใช้ของที่ไซต์
เก็บไว้แล้วเท่านั้น) ส่วน `AGSLOG` อ่านบันทึกที่ไซต์เก็บไว้แล้วเช่นกัน ไม่ได้สั่งให้บันทึกเพิ่มหรือลบ
บันทึกเก่าทิ้งแต่อย่างใด

## การตั้งค่า

### `Config\settings.json`

ชื่อหน่วยงาน ผู้ตรวจสอบ และ **เกณฑ์เตือนทุกค่า** อยู่ในไฟล์นี้ที่เดียว
ไม่มีเกณฑ์ไหนถูกเขียนตายไว้ในสคริปต์ แก้ที่นี่แล้วมีผลทันที

```json
"DiskFreePercent": { "Warn": 20, "Crit": 10, "Direction": "Lower" }
```

`Direction` บอกว่าค่ามาก/น้อยแบบไหนคือแย่ — `Lower` คือยิ่งน้อยยิ่งแย่ (พื้นที่ว่าง, วันคงเหลือก่อนหมดอายุ)
ส่วน `Higher` คือยิ่งมากยิ่งแย่ (การใช้งาน CPU, จำนวนวันที่ไม่ได้ patch)

**หัวข้อข้อเสนอแนะ** — ปิดไว้เป็นค่าเริ่มต้นเพื่อให้รายงานกระชับ เปิดกลับได้ที่

```json
"Report": { "ShowRecommendations": true }
```

เมื่อปิด รายงานจะเหลือ 3 ส่วน คือ สรุปผลสำหรับผู้บริหาร / ภาพรวมรายหัวข้อ / รายละเอียดผลการตรวจสอบ
โดยช่อง "สรุปย่อ" ในตารางภาพรวมยังบอกอยู่ว่าแต่ละหัวข้อพบอะไร
และ **ข้อเสนอแนะฉบับเต็มยังถูกเก็บไว้ใน `PM-Data.json` ครบเสมอ** ไม่ว่าจะเปิดหรือปิด
จึงย้อนไปดูได้ตลอดโดยไม่ต้องรันใหม่

### `Config\services.json`

ใส่ชื่อบริการสำคัญของระบบนั้น ๆ ลงใน `Watchlist` ถ้าบริการในรายการหยุดทำงานจะถูกจัดเป็น **วิกฤต**

ถ้าปล่อยว่างไว้ก็ใช้งานได้ทันที เพราะเครื่องมือจะตรวจหาบริการที่ตั้งค่าเริ่มอัตโนมัติ
แต่หยุดอยู่ให้เองอยู่แล้ว (จัดเป็นระดับควรเฝ้าระวัง) ในไฟล์มีตัวอย่างชื่อบริการที่พบบ่อยแยกตาม role ให้คัดลอกไปใช้

> ใส่ **ชื่อบริการ** ไม่ใช่ชื่อที่แสดง ดูชื่อจริงได้ด้วย
> `Get-Service | Select-Object Name, DisplayName, Status`

### `Config\i18n.json`

ข้อความทุกคำที่ปรากฏในรายงานอยู่ในไฟล์นี้ ทั้งภาษาไทยและอังกฤษ
ถ้าต้องการเปลี่ยนสำนวนให้ตรงกับที่หน่วยงานใช้ แก้ที่นี่ได้เลยโดยไม่ต้องแตะสคริปต์
เครื่องหมาย `{0}` `{1}` คือช่องเติมค่า **ห้ามลบหรือสลับลำดับ**

## การเพิ่มหัวข้อตรวจใหม่

โครงสร้างเป็นแบบ **1 ไฟล์ = 1 หัวข้อ** เพิ่มหัวข้อใหม่ได้โดยไม่ต้องแก้ไฟล์เดิมเลย

1. สร้างไฟล์ใหม่ใน `Checks\` ตั้งชื่อขึ้นต้นด้วยตัวเลขเพื่อกำหนดลำดับ เช่น `98-MyCheck.ps1`
2. เขียนฟังก์ชันที่คืนค่าด้วย `New-PMResult` (ดูรูปแบบจาก `20-DiskSpace.ps1` เป็นตัวอย่างที่สั้นที่สุด)
3. ปิดท้ายไฟล์ด้วย `Register-PMCheck -Id ... -TitleKey ... -Function ...`
4. เพิ่มข้อความไทย/อังกฤษที่ใช้ลงใน `Config\i18n.json`

ตัวสร้างรายงานไม่จำเป็นต้องรู้จักหัวข้อใหม่เลย เพราะทุกหัวข้อคืนค่าโครงสร้างเดียวกัน
เหตุผลเชิงออกแบบเบื้องหลังโครงสร้างนี้ดูเพิ่มได้ใน [HANDOVER.md](HANDOVER.md)
