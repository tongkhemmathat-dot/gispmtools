# PMtools — เครื่องมือบำรุงรักษาเชิงป้องกัน Windows Server

เก็บสถานะเครื่องแม่ข่ายทุกหัวข้อที่ควรรู้ในรอบเดียว แล้วออกไฟล์รายงาน HTML สองภาษา (ไทย/อังกฤษ)
ที่ส่งให้ผู้บริหารหรือหน่วยงานราชการได้โดยตรง หรือสั่ง Print to PDF แนบหนังสือได้ทันที

### เอกสารในชุดนี้มี 3 ไฟล์

| ไฟล์ | สำหรับใคร | เนื้อหา |
|---|---|---|
| **[Manual.html](Manual.html)** | **ผู้ใช้งานทั่วไป** | คู่มือการใช้งานฉบับเต็ม เปิดในเบราว์เซอร์ มีสารบัญ ภาพประกอบ พิมพ์เป็น PDF ได้ — **ส่งต่อให้เพื่อนร่วมงานได้เลย** |
| `README.md` (ไฟล์นี้) | ผู้ดูแลเครื่องมือ | สรุปการใช้งานและการตั้งค่าแบบข้อความ อ่านเร็ว |
| [HANDOVER.md](HANDOVER.md) | ผู้พัฒนาต่อ | สถานะงาน สิ่งที่ค้าง เหตุผลของการตัดสินใจ และกฎที่ห้ามผิดเวลาแก้โค้ด |

> **ถ้ากลับมาทำงานต่อหรือจะแก้โค้ด ให้อ่าน `HANDOVER.md` ก่อน**

## อ่านค่าอย่างเดียว ไม่แก้ไขระบบ

เครื่องมือนี้ **ไม่เปลี่ยนค่าใด ๆ บนเครื่องแม่ข่าย** ทุกหัวข้อเป็นการอ่านสถานะเท่านั้น

- ไม่ตั้งค่า ไม่แก้ registry ไม่แตะ Group Policy
- ไม่เริ่ม ไม่หยุด ไม่แก้ไขบริการใด ๆ
- ไม่ติดตั้งอะไร ไม่ดาวน์โหลดอะไร
- **ไม่รีสตาร์ทเครื่อง** และไม่สั่งอะไรที่ทำให้บริการสะดุด
- สิ่งเดียวที่เขียนลงดิสก์คือโฟลเดอร์รายงานใน `Output\`

**สองหัวข้อที่ปิดไว้เป็นค่าเริ่มต้น** เพราะเป็นเพียงสองหัวข้อที่ติดต่อออกนอกเครื่อง

| รหัส | ทำอะไร | ทำไมถึงปิดไว้ | เปิดใช้เฉพาะครั้ง |
|---|---|---|---|
| `WU` | ถาม WSUS/Microsoft Update ว่ามีอะไรค้างติดตั้ง | ใช้เวลานานมาก และค้างได้บนเครือข่ายปิด (ทดสอบแล้วค้างเกิน 10 นาที) | `-Only WU` |
| `CONN` | ping ไปยัง gateway และ DNS ทุกตัว | ถ้าหน่วยงานมี IPS/IDS อาจมองว่าเป็นการสแกนแล้วบล็อกเครื่อง | `-Only CONN` |

อีก 13 หัวข้อที่เหลือเป็นการอ่านค่าในเครื่องล้วน ๆ ไม่ส่งแพ็กเก็ตออกไปเลย

## วิธีใช้

1. คัดลอกโฟลเดอร์ `PMtools` ทั้งโฟลเดอร์ไปวางบนเครื่องแม่ข่าย
2. เปิด `Config\settings.json` แก้ชื่อหน่วยงานและชื่อผู้ตรวจสอบ (ทำครั้งเดียว)
3. ดับเบิลคลิก **`Run-PM.cmd`** แล้วกดอนุญาตเมื่อระบบขอสิทธิ์ผู้ดูแลระบบ
4. **เลือกจากเมนู** ว่าจะเก็บข้อมูล CPU/หน่วยความจำก่อนกี่นาที (กด Enter เฉย ๆ = ไม่เก็บ ตรวจเลย)
5. รอจนเสร็จ รายงานจะเปิดขึ้นเองในเบราว์เซอร์
6. ในเบราว์เซอร์กด `Ctrl + P` แล้วเลือก **Save as PDF** เพื่อได้ไฟล์แนบหนังสือ

### หน้าตาเมนู

```
==============================================================
  PMtools 1.0.0 - Preventive Maintenance
  Server: SRV-APP01      2026-07-22 11:15
==============================================================

  Existing sample data: 180 samples, collected 09:33 on 2026-07-22
                        option 1 will still chart it.

  Collect CPU / memory samples before the assessment?

   [1]  No sampling - assess now                 (about 10 s)
   [2]  Sample 15 minutes, then assess           (15 min, finishes about 11:30)
   [3]  Sample 30 minutes, then assess           (30 min, finishes about 11:45)
   [4]  Sample 60 minutes, then assess           (60 min, finishes about 12:15)
   [5]  Sample for a custom duration...
   [6]  Sample only - do not build a report

   [Q]  Quit

  Choice [1]:
```

- แต่ละตัวเลือกบอก **เวลาที่จะเสร็จ** ให้เลย จะได้กะเวลาถูก
- ถ้าเคยเก็บข้อมูลไว้แล้วและยังไม่เกิน 24 ชั่วโมง เมนูจะบอกให้ทราบ — เลือก `[1]` ก็ได้กราฟอยู่ดี ไม่ต้องเก็บซ้ำ
- ตัวเลือก `[6]` เก็บอย่างเดียวไม่สร้างรายงาน เหมาะกับเก็บช่วงที่ระบบมีภาระงานจริงไว้ก่อน แล้วค่อยทำรายงานทีหลัง
- **กด Enter เฉย ๆ = ตัวเลือก 1** จงใจให้เป็นแบบนี้ กด Enter พลาดจะได้ไม่ติดอยู่กับการรอ 30 นาที
- เมนูเป็นภาษาอังกฤษเพราะหน้าจอ console ของ Windows Server ส่วนใหญ่แสดงภาษาไทยเป็นกล่องหรือ `???`
  (คนละเรื่องกับตัวรายงาน HTML ซึ่งเป็นภาษาไทยเต็มรูปแบบ)

ถ้าไม่ต้องการเมนู ให้สั่ง `Start-PMCheck.ps1` ตรง ๆ ตามหัวข้อถัดไป

ผลลัพธ์อยู่ที่ `Output\<ชื่อเครื่อง>_<วันเวลา>\`

| ไฟล์ | ใช้ทำอะไร |
|---|---|
| `PM-Report.html` | ไฟล์รายงานที่ส่งให้ผู้ใช้ เปิดได้ทุกเครื่อง ไม่ต้องต่อเน็ต |
| `PM-Data.json` | ข้อมูลดิบทั้งหมด เก็บไว้เทียบย้อนหลังหรือส่งต่อระบบอื่น |

ส่วนไฟล์ตัวอย่าง CPU/หน่วยความจำอยู่ที่ `Output\_Perf\` แยกไว้ต่างหาก เพราะใช้ร่วมกันได้หลายรายงาน

### สั่งจาก PowerShell (ถ้าต้องการควบคุมเพิ่ม)

```powershell
.\Start-PMCheck.ps1                    # รันชุดปกติ
.\Start-PMCheck.ps1 -OpenReport        # รันแล้วเปิดรายงานให้เลย
.\Start-PMCheck.ps1 -Only DISK,CERT    # รันเฉพาะบางหัวข้อ
.\Start-PMCheck.ps1 -Skip CONN         # รันทุกหัวข้อยกเว้นที่ระบุ
.\Start-PMCheck.ps1 -Only WU           # ตรวจรายการปรับปรุงที่ค้าง (ช้า ใช้เมื่อจำเป็น)
```

**ควรรันด้วยสิทธิ์ผู้ดูแลระบบ** ถ้ารันโดยไม่มีสิทธิ์ เครื่องมือจะไม่หยุดทำงาน
แต่บางหัวข้อจะได้ข้อมูลไม่ครบ และจะมีข้อความเตือนแสดงไว้ในรายงานอย่างชัดเจน

## แจกแบบไฟล์เดียว

การคัดลอกทั้งโฟลเดอร์ 24 ไฟล์ไปเครื่องแม่ข่ายมักเป็นขั้นตอนที่เสียเวลาที่สุด
โดยเฉพาะเวลาส่งทางอีเมลหรือผ่านเครื่องที่เข้าถึงได้จำกัด
`Build-PMSingle.ps1` จะรวมทุกอย่างเป็น **ไฟล์เดียว** ที่แตกตัวเองได้

```powershell
.\Build-PMSingle.ps1              # ได้ Dist\PMtools-1.0.0.cmd (~94 KB)
.\Build-PMSingle.ps1 -Format Both # ได้ทั้ง .cmd และ .ps1
```

จากนั้นคัดลอกไฟล์นั้นไฟล์เดียวไปเครื่องแม่ข่าย **ไม่ต้องเอาอะไรไปด้วย**

| ทำอะไร | สั่งอย่างไร |
|---|---|
| ตรวจตามปกติ | ดับเบิลคลิก `PMtools-1.0.0.cmd` (ขอสิทธิ์ผู้ดูแลระบบให้เอง แล้วขึ้นเมนูเดิม) |
| ตรวจเฉพาะหัวข้อ | `PMtools-1.0.0.cmd -Only DISK,CERT` |
| แก้ค่าตั้งต้น | `PMtools-1.0.0.cmd -ExtractConfig` แล้วแก้ `Config\settings.json` ที่ได้มา |

โฟลเดอร์รายงาน `<ชื่อเครื่อง>_<วันเวลา>\` ออก**ข้างตัวไฟล์โดยตรง** ไม่มีชั้น `Output\` คั่นเหมือนแบบโฟลเดอร์
เพราะตัวโปรแกรมอยู่ในโฟลเดอร์ชั่วคราวที่ถูกลบทิ้งหลังรันเสร็จ วางไฟล์ไว้ที่ไหน รายงานก็โผล่ตรงนั้น

```
D:\ตรวจเครื่อง\
    PMtools-1.0.0.cmd
    SRV-APP01_20260722-1305\      <- รอบนี้
        PM-Report.html
        PM-Data.json
    SRV-APP01_20260721-0930\      <- รอบก่อน แยกโฟลเดอร์ ไม่ทับกัน
```

ถ้าที่วางไฟล์เขียนไม่ได้ เช่นวางบน share แบบอ่านอย่างเดียวหรือ USB ที่ล็อกไว้
จะย้ายไปเขียนที่โฟลเดอร์ชั่วคราวแทนและแจ้งให้ทราบ — ตรวจด้วยการลองเขียนไฟล์จริง
ไม่ใช่แค่ดูว่าโฟลเดอร์มีอยู่ไหม เพราะต้องรู้ตั้งแต่ก่อนตรวจ ไม่ใช่ตอนจะเขียนรายงาน

**ยังอ่านค่าอย่างเดียวเหมือนเดิม** ไฟล์นี้เป็นแค่ภาชนะ ข้างในคือสคริปต์ชุดเดิมทุกบรรทัด
ตอนรันมันจะแตกตัวเองลงโฟลเดอร์ชั่วคราว เรียกสคริปต์เดิม แล้วลบโฟลเดอร์นั้นทิ้งเมื่อจบ
(รวมถึงตอนที่มีหัวข้อใดพัง) สิ่งเดียวที่เหลือไว้คือโฟลเดอร์รายงาน

### เรื่องที่ควรรู้

- **`.cmd` แก้ปัญหา execution policy ให้เอง** — มันเรียก `powershell -ExecutionPolicy Bypass`
  ให้เฉพาะ process ตัวเอง ไม่ได้แก้ค่าอะไรบนเครื่อง และ `.cmd` คลิกขวาสั่ง
  *Run as administrator* ได้ ซึ่ง `.ps1` ทำไม่ได้
- **`.ps1` มีไว้สำหรับ Task Scheduler หรือ pipeline** ที่เรียก `powershell.exe` เองอยู่แล้ว
  และไม่ต้องการหน้าต่างขอสิทธิ์ — สั่งด้วย `powershell -ExecutionPolicy Bypass -File PMtools-1.0.0.ps1`
- **ถ้าส่งไฟล์ผ่านอีเมลหรือดาวน์โหลดมา** Windows จะติดเครื่องหมายว่ามาจากเครื่องอื่น
  ให้คลิกขวาที่ไฟล์ → Properties → ติ๊ก **Unblock** ก่อนรัน
- **ไฟล์ที่ได้ห้ามแก้มือ** ให้แก้ที่ซอร์สแล้ว build ใหม่ — ไม่มีขั้นตอนแปลงโค้ดใด ๆ
  ระหว่าง build ดังนั้นเพิ่มหัวข้อตรวจใหม่แล้ว build ซ้ำได้เลย ไม่ต้องแตะ `Build-PMSingle.ps1`
- ข้อความภาษาไทยใน `Config\*.json` ปลอดภัย เพราะ payload เก็บเป็น base64 ซึ่งตรงทุกไบต์

## กราฟแนวโน้ม CPU / หน่วยความจำ

หัวข้อ `PERF` เป็นการอ่านค่า ณ วินาทีที่ตรวจ ซึ่งตอบไม่ได้ว่าภาระงานสูงต่อเนื่องหรือแค่กระชากชั่วครู่
ถ้าต้องการคำตอบนั้นให้เก็บตัวอย่างต่อเนื่องก่อน แล้วกราฟจะขึ้นในรายงานเอง

```powershell
# วิธีที่ 1 — เก็บ 30 นาทีแล้วสร้างรายงานในคำสั่งเดียว
.\Start-PMCheck.ps1 -MonitorMinutes 30 -OpenReport

# วิธีที่ 2 — แยกขั้นตอน (เก็บทิ้งไว้ก่อน ค่อยสร้างรายงานทีหลัง)
.\Start-PMMonitor.ps1 -Minutes 30
.\Start-PMCheck.ps1

# ปรับความถี่ได้ เช่น เก็บ 5 นาที ทุก 2 วินาที
.\Start-PMMonitor.ps1 -Minutes 5 -IntervalSeconds 2
```

ระหว่างเก็บจะมีแถบความคืบหน้าแสดงจำนวนตัวอย่างและค่าล่าสุด **กด `Ctrl+C` หยุดกลางคันได้**
ตัวอย่างที่เก็บไว้แล้วจะไม่หาย เพราะเขียนลงไฟล์ทุกครั้งที่เก็บ

- ค่าที่ได้เก็บเป็น CSV ที่ `Output\_Perf\` เปิดด้วย Excel ได้โดยตรงถ้าอยากทำกราฟเอง
- ตอนสร้างรายงาน หัวข้อ `TREND` จะหยิบไฟล์ล่าสุดมาใช้ ถ้าไฟล์เก่าเกิน 24 ชั่วโมงจะไม่นำมาแสดง
  เพราะไม่สะท้อนสภาพปัจจุบัน (ปรับได้ที่ `Monitor.MaxDataAgeHours`)
- ถ้ายังไม่เคยเก็บตัวอย่าง หัวข้อนี้จะขึ้นเป็น "ข้อมูล" พร้อมบอกวิธีเก็บ ไม่ถือเป็นข้อผิดพลาด

**การตัดสินสถานะใช้เปอร์เซ็นไทล์ที่ 95 ไม่ใช่ค่าสูงสุด** เพราะการกระชากถึง 100% เพียงวินาทีเดียว
เป็นเรื่องปกติของเครื่องแม่ข่ายทุกเครื่อง แต่ถ้าเปอร์เซ็นไทล์ที่ 95 อยู่ที่ 90% แปลว่าภาระงานสูงจริงเกือบตลอดเวลา
ตารางแสดงทั้ง ต่ำสุด / เฉลี่ย / เปอร์เซ็นไทล์ที่ 95 / สูงสุด ให้เห็นครบ

**ต้นทุนการเก็บ** — แต่ละตัวอย่างใช้เวลาประมาณ 0.4 วินาที ที่ความถี่ 10 วินาทีจึงใช้ทรัพยากรราว 4% ของหนึ่งคอร์
เป็นการเฝ้าดูภาระงานโดยแทบไม่เพิ่มภาระเอง

## หัวข้อที่ตรวจ

| รหัส | หัวข้อ | เกณฑ์เตือน (ปรับได้ใน `settings.json`) |
|---|---|---|
| `SYSTEM` | ข้อมูลระบบและเวลาทำงานต่อเนื่อง | เปิดต่อเนื่องเกิน 90 วัน |
| `DISK` | พื้นที่จัดเก็บข้อมูล | ว่าง < 20% หรือ < 20 GB (วิกฤตที่ 10%/10 GB) |
| `PERF` | หน่วยประมวลผลและหน่วยความจำ (ค่า ณ ขณะตรวจ) | CPU > 80% หรือหน่วยความจำว่าง < 15% |
| `TREND` | กราฟแนวโน้มตามช่วงเวลา | เปอร์เซ็นไทล์ที่ 95 เกินเกณฑ์เดียวกับ `PERF` (ต้องเก็บตัวอย่างก่อน) |
| `PROC` | โปรเซสที่ใช้ทรัพยากรสูงสุด | ข้อมูลประกอบ ไม่มีเกณฑ์ |
| `SVC` | สถานะบริการ | บริการอัตโนมัติที่หยุดอยู่ / บริการใน watchlist ที่หยุด |
| `EVT` | บันทึกเหตุการณ์ | มี Error หรือ Critical ในช่วง 7 วัน |
| `HOTFIX` | ประวัติการติดตั้งรายการปรับปรุง | ไม่ได้ patch เกิน 60 วัน |
| `AV` | ระบบป้องกันไวรัส | ปิดการป้องกันเรียลไทม์ / ฐานข้อมูลเก่าเกิน 3 วัน |
| `BACKUP` | การสำรองข้อมูล | สำรองล่าสุดเกิน 7 วัน หรือมีรายการล้มเหลว |
| `CERT` | ใบรับรองอิเล็กทรอนิกส์ | หมดอายุใน 30 วัน (วิกฤตที่ 7 วัน) |
| `TIME` | การเทียบเวลามาตรฐาน | เครื่องไม่ได้ sync เวลา |
| `NET` | อุปกรณ์เครือข่ายและการตั้งค่า | ไม่มีการ์ดใดกำหนด DNS หรือ Gateway เลย |
| `CONN` | การเชื่อมต่อเครือข่าย | ping ไม่ผ่าน — **ปิดไว้เป็นค่าเริ่มต้น** |
| `REBOOT` | สถานะการรอเริ่มระบบใหม่ | มีรายการค้างรอรีสตาร์ท |
| `WU` | รายการปรับปรุงที่ค้างติดตั้ง | **ปิดไว้เป็นค่าเริ่มต้น** |

## การตั้งค่า

### `Config\settings.json`

ชื่อหน่วยงาน ผู้ตรวจสอบ และ **เกณฑ์เตือนทุกค่า** อยู่ในไฟล์นี้ที่เดียว
ไม่มีเกณฑ์ไหนถูกเขียนตายไว้ในสคริปต์ แก้ที่นี่แล้วมีผลทันที

```json
"DiskFreePercent": { "Warn": 20, "Crit": 10, "Direction": "Lower" }
```

`Direction` บอกว่าค่ามาก/น้อยแบบไหนคือแย่ — `Lower` คือยิ่งน้อยยิ่งแย่ (พื้นที่ว่าง, วันคงเหลือก่อนหมดอายุ)
ส่วน `Higher` คือยิ่งมากยิ่งแย่ (การใช้งาน CPU, จำนวนวันที่ไม่ได้ patch)

**หัวข้อข้อเสนอแนะ** — ปิดไว้เป็นค่าเริ่มต้นเพื่อให้รายงานกระชับ เปิดกลับได้ที่

```json
"Report": { "ShowRecommendations": true }
```

เมื่อปิด รายงานจะเหลือ 3 ส่วน คือ สรุปผลสำหรับผู้บริหาร / ภาพรวมรายหัวข้อ / รายละเอียดผลการตรวจสอบ
โดยช่อง "สรุปย่อ" ในตารางภาพรวมยังบอกอยู่ว่าแต่ละหัวข้อพบอะไร
และ **ข้อเสนอแนะฉบับเต็มยังถูกเก็บไว้ใน `PM-Data.json` ครบเสมอ** ไม่ว่าจะเปิดหรือปิด
จึงย้อนไปดูได้ตลอดโดยไม่ต้องรันใหม่

### `Config\services.json`

ใส่ชื่อบริการสำคัญของระบบนั้น ๆ ลงใน `Watchlist` ถ้าบริการในรายการหยุดทำงานจะถูกจัดเป็น **วิกฤต**

ถ้าปล่อยว่างไว้ก็ใช้งานได้ทันที เพราะเครื่องมือจะตรวจหาบริการที่ตั้งค่าเริ่มอัตโนมัติ
แต่หยุดอยู่ให้เองอยู่แล้ว (จัดเป็นระดับควรเฝ้าระวัง) ในไฟล์มีตัวอย่างชื่อบริการที่พบบ่อยแยกตาม role ให้คัดลอกไปใช้

> ใส่ **ชื่อบริการ** ไม่ใช่ชื่อที่แสดง ดูชื่อจริงได้ด้วย
> `Get-Service | Select-Object Name, DisplayName, Status`

### `Config\i18n.json`

ข้อความทุกคำที่ปรากฏในรายงานอยู่ในไฟล์นี้ ทั้งภาษาไทยและอังกฤษ
ถ้าต้องการเปลี่ยนสำนวนให้ตรงกับที่หน่วยงานใช้ แก้ที่นี่ได้เลยโดยไม่ต้องแตะสคริปต์
เครื่องหมาย `{0}` `{1}` คือช่องเติมค่า **ห้ามลบหรือสลับลำดับ**

## หมายเหตุทางเทคนิค

**เรื่อง encoding** — PowerShell 5.1 อ่านไฟล์ `.ps1` เป็น ANSI ถ้าไฟล์ไม่มี BOM
ภาษาไทยที่เขียนตรง ๆ ในสคริปต์จึงเพี้ยน ด้วยเหตุนี้ไฟล์ `.ps1` ทุกไฟล์ในเครื่องมือนี้จึงเป็น
**ASCII ล้วน** และภาษาไทยทั้งหมดถูกย้ายไปอยู่ในไฟล์ JSON ที่อ่านด้วย `-Encoding UTF8` แทน
ถ้าจะแก้สคริปต์เพิ่มเอง **อย่าพิมพ์ภาษาไทยลงในไฟล์ `.ps1`** ให้เพิ่มคีย์ใน `i18n.json` แทน
(คำสั่งตรวจว่าทุกไฟล์ยังเป็น ASCII อยู่ ดูใน [HANDOVER.md](HANDOVER.md) หัวข้อ "กฎที่ห้ามผิดเวลาแก้โค้ด")

**เรื่องปี พ.ศ.** — หัวรายงานแสดงวันที่เป็น พ.ศ. คู่กับ ค.ศ.
ส่วนวันที่ในตารางใช้รูปแบบ `yyyy-MM-dd` เป็น **ค.ศ.** เพราะช่องตารางมีค่าเดียวแต่มีผู้อ่านสองภาษา
ถ้าใส่ 2569 โดยไม่กำกับจะกำกวมสำหรับผู้อ่านอีกฝั่ง

**เรื่องภาษาของ Windows** — เครื่องมือหลีกเลี่ยงการอ่านค่าที่แปลตามภาษาเครื่อง

- อ่าน CPU ผ่านคลาส `Win32_PerfFormattedData_PerfOS_Processor` แทน `Get-Counter`
  เพราะ**ชื่อคลาส WMI เป็นภาษาอังกฤษเสมอ** ส่วนชื่อ performance counter ถูกแปล
  (`'\Processor(_Total)\% Processor Time'` ใช้ไม่ได้บน Windows ที่ไม่ใช่ภาษาอังกฤษ)
- จัดกลุ่ม event ด้วยเลข `Level` แทน `LevelDisplayName` ซึ่งถูกแปล
- หัวข้อ `TIME` อ่านค่าจากลำดับบรรทัดและตัวเลขของ `w32tm` ไม่ใช่จากข้อความ ซึ่งถูกแปลเช่นกัน

จึงให้ผลเหมือนกันทั้งบน Windows ภาษาอังกฤษและภาษาไทย

**เรื่องเครือข่าย — สาเหตุที่ไม่ใช้ `Get-NetAdapter` และ `Get-NetIPConfiguration`**

เคยพบว่าระหว่างรันแล้วเครือข่ายของเครื่องไม่เสถียร (RDP ค้างชั่วขณะ, latency สูงขึ้น)
โดยเฉพาะบนเครื่องที่ทำ **NIC Teaming** สาเหตุคือสองคำสั่งนั้นไม่ได้อ่านค่าจากในเครื่องเฉย ๆ

- `Get-NetAdapter` เรียกผ่าน WMI provider ของ **ไดรเวอร์การ์ดเครือข่าย** เมื่อมี teaming ตัว LBFO
  ต้องไล่เก็บค่าจากการ์ดสมาชิกทุกใบ ซึ่งบล็อกสแตกเครือข่ายชั่วครู่ได้
- `Get-NetIPConfiguration` เรียก **Network List Service** เพิ่มอีก ซึ่งอาจกระตุ้นให้ Windows
  ตรวจจับโปรไฟล์เครือข่ายใหม่ ถ้าโปรไฟล์สลับ Domain ↔ Public โปรไฟล์ไฟร์วอลล์จะเปลี่ยนตาม

ตอนนี้หัวข้อ `NET` และ `CONN` จึงอ่านผ่าน **IP Helper API ของ .NET**
(`System.Net.NetworkInformation`) แทน ซึ่งเป็นข้อมูลในเครื่องที่ไม่แตะไดรเวอร์และไม่แตะ NLA
ผลพลอยได้คือเร็วขึ้นจาก 3,048 มิลลิวินาที เหลือ 194 มิลลิวินาที และยังใช้ได้บน
Windows Server 2008 R2 ซึ่งไม่มีคำสั่งตระกูล `Get-Net*` ให้ใช้ตั้งแต่แรก

หากยังพบอาการอยู่ ให้ลองตัดหัวข้อเครือข่ายออกทั้งหมดเพื่อยืนยัน

```powershell
.\Start-PMCheck.ps1 -Skip NET
```

**ยังไม่ได้ทดสอบบน Windows Server จริง** — พัฒนาและทดสอบบน Windows 11 + PowerShell 5.1
จุดที่ควรดูเป็นพิเศษเมื่อนำไปวางบน server ครั้งแรก

- `root\SecurityCenter2` **ไม่มีบน Windows Server** หัวข้อ `AV` จึงมีทางสำรอง 3 ชั้น
  (`Get-MpComputerStatus` → `SecurityCenter2` → บริการ `WinDefend`) และถ้าหาไม่เจอจริง ๆ จะรายงานตรง ๆ ว่าตรวจไม่พบ
- บันทึก `Microsoft-Windows-Backup` มีเฉพาะเมื่อติดตั้ง Windows Server Backup แล้วเท่านั้น
- ถ้าใช้ระบบสำรองข้อมูลยี่ห้ออื่น ให้เพิ่มคำค้นใน `Backup.TaskKeywords` ใน `settings.json`
- หัวข้อ `CONN` รายงาน ping ไม่ผ่านเป็นระดับ *ควรเฝ้าระวัง* ไม่ใช่วิกฤต เพราะ Gateway และ DNS
  จำนวนมากถูกตั้งค่าไม่ให้ตอบ ping ตามนโยบายความมั่นคงปลอดภัย ซึ่งไม่ใช่ความผิดปกติ

## การเพิ่มหัวข้อตรวจใหม่

โครงสร้างเป็นแบบ **1 ไฟล์ = 1 หัวข้อ** เพิ่มหัวข้อใหม่ได้โดยไม่ต้องแก้ไฟล์เดิมเลย

1. สร้างไฟล์ใหม่ใน `Checks\` ตั้งชื่อขึ้นต้นด้วยตัวเลขเพื่อกำหนดลำดับ เช่น `98-MyCheck.ps1`
2. เขียนฟังก์ชันที่คืนค่าด้วย `New-PMResult` (ดูรูปแบบจาก `20-DiskSpace.ps1` เป็นตัวอย่างที่สั้นที่สุด)
3. ปิดท้ายไฟล์ด้วย `Register-PMCheck -Id ... -TitleKey ... -Function ...`
4. เพิ่มข้อความไทย/อังกฤษที่ใช้ลงใน `Config\i18n.json`

ตัวสร้างรายงานไม่จำเป็นต้องรู้จักหัวข้อใหม่เลย เพราะทุกหัวข้อคืนค่าโครงสร้างเดียวกัน
